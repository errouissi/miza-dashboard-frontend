Frontend Implementation Roadmap

Four approved documents say what to build, how it should look, and how it must be organized. None of them says in what order, and order is where projects of this shape actually fail. This document sequences the work: what unblocks what, what can run in parallel, what must be proven before it is generalized, and what "done" means at each step.

Version 1.0 — proposed Constraints Discovery ✅ · Domain Architecture ✅ · Design System v1.0 ✅ · Frontend Technical Architecture v1.0 ✅ Status Awaiting approval before implementation begins

How to read this document

The four approved documents are frozen constraints. This roadmap does not revisit a single architectural decision; where it appears to, it is wrong and they win. References are cited as Discovery, Architecture §n, Design System §n, and FTA §n / D-n (the Frontend Technical Architecture and its numbered decisions).

What this document does decide is execution: milestone boundaries, sequencing, parallelism, risk mitigation, and gates. Where it recommends an order that differs from the one an approved document sketched, that difference is called out explicitly in a ⚖ Decision callout and reconciled — never applied silently.

01

Roadmap philosophy

Six principles govern the ordering. Every sequencing decision in this document is derivable from them, and where two of them conflict, the earlier one wins.

1 · Build the riskiest unknown early, not the easiest win

Sequence by where the project could fail, not by what demos well. The things that can genuinely derail this build are: a missing backend controller, an abstraction that hardens wrongly, a cross-domain cache bug that shows an operator a stale capacity. Each is scheduled to surface as early as it possibly can.

Why: a risk discovered in month four costs the four months. The purpose of ordering is to convert unknowns into knowns while there is still time to react cheaply.

2 · Prove a pattern on real work before generalizing it

No shared abstraction is built before the concrete cases that justify it. The pattern library emerges from the first real resources; it is not authored in advance against an imagined one.

Why: FTA D-11 names the failure mode precisely — "the abstraction that eats the app" — and FTA §12's rule of three exists to prevent it. A roadmap that builds all six page patterns before shipping a single real screen violates the architecture it is meant to execute. This is the single biggest change this document makes to the sketch in Architecture §11, and §2 argues it fully.

3 · Every milestone ships something an operator could use

Milestones are vertical, not horizontal. "The API layer is done" is not a milestone; "villes can be created, listed, edited and deleted by a permitted user against the real backend" is.

Why: horizontal layers cannot be validated — a "finished" data layer with no screen on top of it is a hypothesis. Vertical slices are the only unit of work that proves the layers actually fit together, and they are the only unit a stakeholder can respond to.

4 · Dependencies dictate order; preference does not

Money attributes transactions to agents; Stock moves inventory between them; Grattage is the seam between the two ledgers. That dependency graph (Architecture §2) is a fact about the business, and it fixes the domain sequence regardless of which domain anyone finds most interesting.

Why: building a domain whose foreign keys point at a domain that does not exist yet means mocking those references — and every mock is a second implementation that must later be removed and re-tested.

5 · Unblock the backend before you are blocked by it

Two backend gaps are already known: Companies and Suppliers have no controllers, and several Overview endpoints are unrouted (Discovery §6, Architecture §5). Neither is discovered by this roadmap — both are scheduled by it, as tickets raised at project start rather than at the moment they stall a developer.

Why: backend work has lead time and a different queue. A dependency raised the week it blocks you is a dependency that blocks you.

6 · Enforcement infrastructure comes before the code it governs

The boundary lint rule (FTA §4), the formatter module (Design System §5/§27), and the permission registry (FTA §6) are built first — before there is any code to violate them.

Why: a boundary rule added after three months of building produces two hundred violations, and two hundred violations get suppressed in a batch, and the rule is now decorative. Retrofitting enforcement is not a smaller version of building it early — it is a different, failing activity.

02

Overall delivery strategy

The conventional shape for this kind of project — foundation, then infrastructure, then shared UI, then domains, then hardening — is mostly right, and it is wrong in one specific and expensive way. This section adopts it with one deliberate modification.

0 · Pre-flight Decisions signed, backend tickets raised, enforcement tooling in place. No product code. Ends the moment a developer could start and be blocked by nothing.
↓
1 · Walking skeleton One real resource, end to end, through every layer of the architecture. The infrastructure and the first screen are built together, because that is the only way to know the infrastructure works.
↓
2 · Pattern hardening The second and third resources. The shared patterns are extracted here, from three real cases — not invented earlier from zero.
↓
3 · Business domains Network → Money → Stock → Grattage, in the dependency order the backend already fixed (Architecture §2). Each domain is now mostly configuration over proven patterns.
↓
4 · Integration The cross-domain surfaces — Agent 360, Client 360, the Overview widget grid — which are only meaningful once the domains they compose exist.
↓
5 · Hardening Accessibility, performance, end-to-end coverage of the money paths, observability, flag cleanup. Not a phase for finishing features — a phase for making finished features trustworthy.

R-1 · The foundation is delivered through a real vertical slice, not before one

Architecture §11's Phase 0 proposes building the five layouts, the component library, the permission registry, and the CRUD generators "against no real domain yet — or against a throwaway/reference resource," shipping nothing user-facing. The instinct is sound: get the reusable machinery in place so every later phase is configuration rather than construction. This roadmap adopts the second option §11 offers, and rejects the first — and the distinction is not cosmetic.

Building the pattern library against no real domain directly contradicts the architecture it is meant to serve. FTA D-11 warns of "the abstraction that eats the app," and FTA §12 requires promotion to shared to be justified by stable shared semantics evidenced across independent uses — with two uses explicitly insufficient. A Phase 0 that authors DataTable, FilterBar, ListPage, DetailPage, DraftLifecyclePage, and the CRUD generator against zero real resources produces exactly the artifact both rules exist to prevent: six abstractions fitted to an imagined caller, hardened before any real one exists, and then bent for months to accommodate the real ones as they arrive. The cost does not appear in Phase 0 — it appears in every phase after it, as friction nobody attributes to the decision that caused it.

Instead: the foundation is built as the byproduct of shipping Villes — the simplest real resource in the product — end to end (M1). That single slice forces every layer into existence against a real API: the HTTP client, the error normalizer, the mapper, the query keys, the permission registry, the URL filter state, the formatters, the shell. Then Secteurs (M2a) proves the second case, and Products (M2b) the third — at which point the ListPage/FormDrawer/DataTable patterns are extracted from three working screens rather than predicted. The rule of three is not just an architectural rule here; it is the roadmap's sequencing mechanism.

What is not deferred, and this is the part that keeps §11's intent intact: the things that cannot be retrofitted are still built first, in M0 — the boundary lint rule, the config validator, the permission registry's shape, the formatter module. Enforcement precedes code (§1, principle 6); abstraction follows evidence (principle 2). Those are different things, and Phase 0 conflated them.

Why this order minimizes risk — the specific mechanism

Each stage above exists to convert one class of unknown into a known, in ascending order of what it would cost to discover late.

Pre-flight converts external unknowns (is the language decided? do the endpoints exist? are the error codes documented?) — the ones with the longest lead time and the least controllable. The walking skeleton converts integration unknowns: does our auth actually work against Sanctum, does the mapper handle the real status vocabulary, does the 401 policy behave when five requests fail at once? These are the unknowns that are cheap to fix in one thin slice and catastrophic to discover in the fifteenth. Pattern hardening converts abstraction unknowns: is this really one shape, or three that look alike? The domain phases convert domain unknowns, in dependency order, so no domain is built against a mock of another. Integration converts composition unknowns. Hardening converts quality unknowns — deliberately last, because it is the only class that can be safely discovered late.

03

Milestones

Nine milestones. Each has a goal, deliverables, dependencies, and exit criteria — and each exit criterion is written to be observable, because a criterion that requires judgment to evaluate will be judged generously by the person who wants to move on.

M0

Pre-flight — nothing blocks a developer

blocking

Remove every external dependency and put the enforcement machinery in place, before one line of product code exists. This is the only milestone that ships no feature, and it is the one most likely to be skipped under pressure — which is why it is a gate.

Deliverables
  • O-1 signed off — the interface language (French-first, Design System §27). Blocking: every label, error message, and empty state written before this is settled is rework.
  • Backend tickets raised — Companies/Suppliers controllers; Overview endpoint routing (chartData, recentActivities, agentsOverview); confirmation of the documented domain error-code catalogue.
  • Boundary lint rule enforcing FTA §4's import matrix, failing CI.
  • Config validator with fail-fast bootstrap; the four environments defined (FTA §14).
  • Test harness — MSW wired, fixtures sourced from the backend's Postman collection (FTA D-15), CI running lint + typecheck + tests.
  • ADR log started; ADR-0001 (token storage) recorded as written in FTA §17.
Dependencies

None. This is the entry point. Two deliverables depend on other people (product sign-off on O-1; the backend team accepting tickets), which is precisely why they are scheduled at the beginning rather than at the moment they bite.

Note: the backend tickets do not need to be delivered to exit M0 — only raised and accepted with an estimate. What must not happen is discovering in M5 that Companies has no API and nobody knew.

Exit criteria
  • A deliberate boundary violation, committed to a branch, fails CI — demonstrated, not assumed.
  • The app fails to boot, loudly, with a missing required env var — demonstrated.
  • O-1 has a written decision from product.
  • Backend tickets exist, are accepted, and have owners.
M1

Walking skeleton — one resource, every layer

highest risk

Ship Villes end to end against the real backend: login → shell → list → filter → create → edit → delete, with permissions, error handling, and URL state. The narrowest possible slice that touches every layer of FTA §2.

Deliverables
  • HTTP client + interceptors + AppError normalization (FTA §7, §11).
  • The 401 session policy, including concurrent-401 collapse (FTA §11) — the gap Discovery found, closed on day one rather than patched later.
  • Query client + staleTime tiers + key factory for one resource (FTA §8).
  • Session store (the single localStorage owner) + permission registry and evaluator + route guard generation (FTA §6, §14).
  • AppShell, sidebar rendering from the registry, breadcrumb.
  • Formatters module — money, dates, phone, identifiers (Design System §5/§27).
  • Villes: list, filter-in-URL, create/edit drawer, delete confirmation. Written directly, not through a generator.
Dependencies & risks

Depends on: M0.

This is where the integration risk lives. Everything unproven about the architecture is proven or disproven here: Sanctum bearer flow, the real pagination envelope, the error-code shape, whether the permission strings in the login payload match the registry's names, whether the French/English status vocabulary behaves as the mapper expects.

Villes is chosen deliberately: lowest business stakes, simplest schema, a real endpoint, and no dependency on any other domain. If the architecture is wrong, it is wrong cheaply here.

Exit criteria
  • A permitted user creates a real ville against the real backend, through the full stack.
  • A user without the permission sees no nav item, no button, and the calm 403 on direct URL entry.
  • A filtered list URL, pasted into a second browser, reproduces the same view (Design System §15).
  • An expired token mid-session produces one clean redirect to login, with the return path preserved — verified with concurrent requests in flight.
  • A forced 500 renders the error state with a support reference; a forced 422 maps to its field.
M2

Pattern hardening — the rule of three, executed

the extraction point

Ship Secteurs and Products, and only then extract the shared ListPage / DataTable / FilterBar / FormDrawer patterns and the resource-definition module from the three working screens.

Deliverables
  • M2a — Secteurs: built by copying Villes. Duplication is expected and correct here (FTA §12: two uses do not justify promotion).
  • M2b — Products: the third case. Now the shape is evidence, not a guess.
  • M2c — Extraction: DataTable (TanStack Table headless), FilterBar, ListPage, FormDrawer, the resource-definition module (FTA §12), StatusBadge, MoneyAmount, EntityChip, ConfirmActionDialog.
  • Villes and Secteurs refactored onto the extracted patterns — which is the proof the extraction is real.
Dependencies & rationale

Depends on: M1.

This milestone looks wasteful — three near-identical screens, then a refactor — and it is the cheapest insurance in the roadmap. The alternative is designing the DataTable's API from imagination and then discovering in M4 that approval queues need a row state it cannot express.

The refactor of Villes and Secteurs is not optional. If the extracted pattern cannot absorb the two screens it was extracted from, it is not a pattern — and finding that out costs an afternoon here instead of a quarter later.

Exit criteria
  • Products ships with no new pattern code — a definition, a mapper, a key factory, a route.
  • Villes and Secteurs run on the extracted patterns with no behavior change.
  • The resource-definition module contains no permission literals, no route paths, no business logic (FTA §12).
  • A fourth resource is estimable in hours, and the team agrees on the number.
M3

Network — the identity graph

foundation domain

Admins, Managers, Commercials, Clients. Every subsequent domain foreign-keys into this one, so it is built before any of them (Architecture §2).

Deliverables
  • Admins, Agents (managers + commercials), Clients: ListPage + DetailPage + forms.
  • CreateWizard — agent onboarding, the only wizard in the product (FTA D-9). Multi-file upload with named slots (Design System §12).
  • EntityChip/entity-picker autocomplete — the one sanctioned infinite query (FTA §8).
  • Client bulk-assign — the first bulk action, exercising the 100-cap and the deliberate all-pages selection step (Design System §14).
Dependencies & risks

Depends on: M2 (patterns), and Reference data for the ville/secteur pickers — which is why Reference was the walking-skeleton vehicle rather than an afterthought.

Risk: the onboarding wizard is the first form complex enough to hurt — ~20 required fields, conditional by role, multi-file. It is also the form where FTA §10's rule ("a lost connection MUST NOT cost a filled wizard its data") is genuinely tested.

Exit criteria
  • An agent is onboarded end to end, with documents, against the real backend.
  • A network failure at the wizard's final step loses nothing.
  • The entity picker is used everywhere a foreign key is chosen; no raw ID entry exists in any form (FTA §7).
M4

Money — the first irreversible actions

first money path

Cheques, Deposits, Debt Payments. The first approval queues, the first irreversible confirmations, and the first cross-domain cache invalidation.

Deliverables
  • Cheques: submission, pending queue, approve / reject / annuler.
  • Deposits: list (filterable by type), validate, reject.
  • Debt Payments: list + submit.
  • ApprovalQueuePage — first instance of the pattern (Architecture §8).
  • The freshness rule, implemented (FTA §8): confirmation dialogs render from a fresh refetch, and refuse the action if the record changed underneath the operator.
  • The invalidation map (FTA D-3) — its first real entries: cheque.approved → agent capacity.
  • Error-code registry entries for the Money domain's 409 families (FTA D-10).
Dependencies & parallel track

Depends on: M3 (transactions attribute to agents).

Parallel track — the DraftLifecyclePage spike. While Money is built, a second developer prototypes the LineItemsEditor and DraftLifecyclePage against Bons (§6, §7). This is the roadmap's main risk mitigation: the hardest pattern in the app is de-risked before the milestone that depends on it four times over.

Exit criteria
  • Approving a cheque updates the agent's capacity without a manual refresh — the invalidation map, proven.
  • A confirmation dialog opened on a stale record refuses to submit and explains why.
  • Two admins approving the same pending cheque: the second sees a clean domain error, not a crash.
  • No mutation retries automatically, anywhere (FTA §11) — verified by review and test.
M5

Stock — one lifecycle, four times

largest domain

Bons → Allocations → Agent Transfers → Agent Stock Returns. Four resources, one DraftLifecyclePage, proven once and configured three times.

Deliverables
  • DraftLifecyclePage + LineItemsEditor, productionized from the M4 spike.
  • All four movement types: draft → lines → irreversible validate, plus cancel.
  • Stock domain error codes registered with recovery paths — including COMMERCIAL_HAS_STOCK_CANNOT_REASSIGN linking to the return flow.
  • Zero-stock reassignment guard surfaced in the Agent edit form (Architecture §6) — a Stock value shown inside a Network form, via public surface only.
  • Companies & Suppliers screens — behind a feature flag, dark until the M0 backend ticket lands (FTA D-12).
Dependencies & risks

Depends on: M3 (agents as counterparties), M4's spike (the lifecycle pattern).

Risk — the Companies/Suppliers blocker. If the backend ticket has not landed, Stock does not stall: the four movement types do not depend on those directory screens. The flag keeps the UI dark and the milestone completes without them, with the flag carrying its removal condition (§7).

Risk — validation is irreversible. This is the milestone where a bug moves real stock. The E2E suite (M8) starts here, not at the end.

Exit criteria
  • All four movement types validate irreversibly, with stock-check errors rendering their registered recovery path.
  • The fourth movement type required no changes to DraftLifecyclePage — the pattern held.
  • Reassigning a commercial holding stock is blocked in the form, with an explanation, not by a round-tripped 422.
M6

Grattage — the seam

cross-domain

Invoices, outstanding obligations, and the restock gate — the one deliberate domain→domain import in the architecture (FTA §13).

Deliverables
  • Grattage invoice list / detail / cancel.
  • Outstanding-obligation view per agent.
  • The restock-gate hook, exported from Grattage's public surface and consumed by Stock's Allocation and Agent-Transfer forms (Architecture §6, FTA §4 mechanism 2).
  • Invalidation map: deposit.validated → grattage invoices settle.
  • Deposit ↔ invoice bidirectional links (Architecture §6).
Dependencies & scope

Depends on: M4 (deposit validation) and M5 (stock capacity). Built last among the domains deliberately — a seam is only meaningful once both sides exist, and building it earlier would mean mocking one of them.

Explicitly not built: admin-create and admin-settle. Both routes hard-403 by backend design (Phase 5.8b / 5.10). No UI invokes them, and no workaround is attempted (Architecture §5).

Exit criteria
  • Validating a reconciliation deposit flips its grattage invoices to settled, visibly, without a refresh.
  • Allocating to a gated commercial surfaces the gate in the form, before submit — not as a post-submit failure.
  • Stock imports exactly one thing from Grattage: the gate hook. Verified by the boundary lint.
M7

Integration — the composed surfaces

Agent 360, Client 360, and the Overview widget grid — everything that composes domains rather than owning them.

Deliverables
  • WorkspacePage pattern; Agent 360 (Network + Money + Stock + Grattage) and Client 360.
  • Overview widget grid: KPI tiles, queues, charts (Architecture §5).
  • Independent panel error boundaries — one dead panel must not blank the other four (FTA §11).
  • Parallel query firing verified — no serial waterfall (FTA §8).
Dependencies & risks

Depends on: M3–M6, all complete. This is the milestone that cannot be pulled forward, and attempting to would mean mocking four domains.

Risk: the Overview's chart/activity endpoints were unrouted at discovery. If the M0 ticket has not landed, Overview ships with the widgets whose endpoints exist and flags the rest — the widget grid was designed for exactly this (Architecture §5).

Exit criteria
  • Agent 360 composes four domains with zero boundary-lint suppressions.
  • Its independent panels fire in parallel — verified in the network panel, not assumed.
  • Killing one panel's endpoint degrades that panel only.
M8

Hardening — making it trustworthy

Not a phase for finishing features. A phase for proving the finished ones deserve an operator's trust.

Deliverables
  • E2E suite on the irreversible money paths (FTA §16) — begun at M5, completed here.
  • Accessibility pass: keyboard-complete flows, focus visibility, contrast, reduced motion (Design System §1, §10).
  • Performance: no serial waterfalls, time-to-populated-table measured per domain (FTA §18).
  • Observability live: error reporting with PII scrubbing, correlation IDs surfaced as support references (FTA §11, §18).
  • Flag cleanup — every expired flag removed with its dead branch (FTA §14).
  • ADR log reconciled with what was actually built.
Dependencies

Depends on: M7.

Why quality is last and this is not a contradiction: the rules (formatters, focus states, error handling, boundaries) are enforced from M0 onward and are never deferred. What is deferred is the audit — the systematic sweep that catches what enforcement missed. Deferring the audit is safe; deferring the rules is not, and this roadmap does neither by accident.

Exit criteria
  • Every irreversible flow has a passing E2E test against a real backend.
  • Every flow is completable by keyboard alone.
  • Zero expired feature flags; zero boundary suppressions without an ADR.
  • No P1 defects open.

05

Domain rollout strategy

The domain order is not a preference and it is not negotiable by taste — it is read directly off the dependency graph the backend already enforces (Architecture §2). This section states the justification, and then answers the one reordering question a reasonable engineer will ask.

OrderDomainWhy here, and not earlier or later
1Reference
villes · secteurs · products		 First, because it is the cheapest real thing that exists. No lifecycle, no workflow, no dependency on any other domain — and every Network form needs its pickers. It is therefore both the lowest-risk vehicle for proving the architecture (M1) and a genuine prerequisite for M3. The two facts point the same way, which is a good sign that this is right.
2Network
admins · agents · clients		 Second, because everything downstream foreign-keys into it. Cheques attribute to an agent; allocations move stock to an agent; grattage invoices belong to a client. Building any of those first means mocking agents — and a mock is a second implementation you must later remove and re-test. Network also contains the one wizard, so its complexity is met while the schedule still has slack.
3Money
cheques · deposits · debt payments		 Third, because it produces the capacity Stock consumes. Approving a cheque grants an agent rapped and grattage capacity; Stock's allocation forms read it. Building Stock first means its capacity displays have nothing real to read. Money is also structurally simpler than Stock — ListPage, DetailPage, ApprovalQueuePage, all proven by M2 — which makes it the right place to meet the first irreversible action, rather than meeting it for the first time inside the hardest pattern in the app.
4Stock
bons · allocations · transfers · returns		 Fourth, and the largest. Four resources sharing one lifecycle pattern — proven once and reused three times, which is why they are built together rather than scattered. It sits after Money because of capacity, and after the M4 spike because DraftLifecyclePage is the riskiest pattern in the product and should not be met cold.
5Grattage
the seam		 Fifth, because a seam requires both sides. It consumes Stock and is settled by Money's deposit validation. Built earlier, its two most important behaviors — the settlement cascade and the restock gate — could only be mocked, and a mocked seam proves nothing. Built here, both are testable end to end on the first day they exist.
6Overview & workspaces
composition		 Last, because it owns nothing. It is a read surface over the five domains above (Architecture §5). It cannot be built before them without mocking all of them, and it is the one part of the app whose scope will expand to fill whatever time remains — which is another reason to bound it at the end rather than let it sprawl at the start.

R-2 · Should Stock come before Money, to meet the hardest pattern earlier?

This is the strongest available objection to the approved order, and it deserves a real answer rather than deference. The argument: DraftLifecyclePage is the app's riskiest pattern, four resources depend on it, and principle 1 of §1 says build the riskiest unknown early. On that logic Stock should precede Money.

Rejected — but the concern is legitimate, and it is addressed rather than dismissed.

Reordering fails on a dependency that is not negotiable: Stock's allocation and transfer forms display and gate on agent capacity, which is produced by Money's cheque approvals. Building Stock first means either mocking capacity (and re-testing it all later against the real thing) or shipping Stock forms whose most important number is fake. Worse, it would mean meeting the product's first irreversible action — a validate that debits one agent and credits another — inside its most complex and least proven UI pattern, simultaneously. Two hard problems at once is not risk reduction; it is risk concentration.

The concern is answered by a spike, not by a reorder. The DraftLifecyclePage and LineItemsEditor are prototyped in parallel during M4 (§6), against Bons, by a second developer. The riskiest pattern is therefore de-risked before M5 starts — which is what principle 1 actually asks for — while the dependency order stays intact and irreversibility is met first in the simpler domain. This is strictly better than reordering on both axes, and it costs nothing but a decision to schedule it.

06

Parallel work opportunities

Parallelism is a multiplier on throughput and a multiplier on merge pain. This section identifies what can genuinely run concurrently without violating a boundary or racing on a shared file — and states plainly where parallelism must not be attempted.

The precondition: parallelism starts at M2, not before

M0 and M1 are single-track work and MUST NOT be parallelized. The walking skeleton is one thin vertical slice through every layer; splitting it across developers means two people negotiating the shape of the HTTP client, the error model, and the query keys while discovering them. The coordination cost exceeds the work, and the output is a compromise architecture assembled by committee in week one.

One developer builds the skeleton. Everyone else works the genuinely independent track below.

Track B — independent from day one

This work has no dependency on the domain layer at all — it is pure, testable, and specified in complete detail by the approved Design System. It can be built by a second developer during M0/M1 and consumed the moment it lands.

Formatters

Money, dates, relative dates, phone, identifiers, percentages, large numbers, absent values (Design System §5, §27; FTA §14). Pure functions, fully specified, unit-testable in isolation. The highest value-per-hour work in the entire project and it depends on nothing.

Presentational primitives

StatusBadge (the six-tone vocabulary, Design System §17), MoneyAmount, EntityChip, the empty-state and error-state components (§22, §23). They take data as props and know no domain — which is exactly what makes them parallelizable.

Test & tooling harness

MSW setup, fixtures derived from the Postman collection (FTA D-15), render helpers, CI pipeline, the boundary lint rule. Infrastructure for everyone, blocking no one.

Hot files — the shared-state problem of a parallel team

Four files in this architecture are touched by every domain, and they are therefore the merge-conflict and correctness hotspots of any parallel work:

(1) The permission registry (FTA §6). (2) The invalidation map (FTA D-3). (3) The error-code registry (FTA D-10). (4) The route table assembly (FTA §5).

They are intentionally central — that centralization is the whole point of D-3, and decentralizing them to reduce merge friction would reverse an approved decision. So the mitigation is procedural, not architectural: these files are append-mostly (new entries, rarely edited ones), each domain's entries are added in that domain's own pull request, and a change that restructures one of them is a standalone PR that lands alone. Textual conflicts on append-only maps are trivial; semantic conflicts on a restructure are not, and the two must never arrive together.

What MUST NOT be parallelized

The walking skeleton (M1) — one architecture, discovered once, by one person, or it is discovered three ways.

The pattern extraction (M2c) — a shared abstraction designed by two people concurrently acquires both of their assumptions and satisfies neither. Extract single-track, then parallelize the consumers.

Two domains at once, before M3 completes. Every domain foreign-keys into Network; starting Money and Stock in parallel before agents exist means both teams mock agents, both build against the mock, and both re-test when the real thing lands. That is not parallel work — it is duplicated work with a delayed merge.

Anything that requires suppressing a boundary lint rule to proceed. If parallel work needs a suppression to compile, the split was wrong. Re-cut the work, do not suppress the rule (FTA §4).

High-risk areas

Ranked by expected cost — probability multiplied by what it would take to recover. Each row states the mitigation already scheduled in this roadmap, because a risk register that does not change the plan is decoration.

Business & correctness risk — the ones that cost money
Stale-cache approval
highest consequence		 An operator approves a cheque or validates a deposit against a value that changed underneath them. This is the failure the entire architecture is organized against, and it produces a financially wrong outcome that looks like a successful one — the worst possible combination, because nothing surfaces it. Fixtures are derived from the backend's maintained Postman collection, not invented. Architectural erosion risk — the slow ones
Premature abstraction
structural		 The pattern library hardens against imagined requirements and is then bent for months (FTA D-11's "abstraction that eats the app"). Nobody attributes the resulting friction to the decision that caused it, so it is never corrected. This is the risk that reshaped the roadmap (R-1). Abstraction is deferred to M2 and requires three real cases. The M2 exit criterion — the third resource ships with no new pattern code, and the first two refactor onto it cleanly — is the objective test that the abstraction is real rather than assumed.
Boundary erosion The shared layer starts importing domain types "just for the props," and within a year the dependency rules of FTA §4 are decorative. The lint rule is an M0 deliverable — before any code exists to violate it. Retrofitting it later produces a batch of suppressions and a dead rule. Suppression count is reviewed at every gate (§8).
The config DSL forms The resource definition accretes callbacks until it is a programming language that is harder to read than the pages it replaced (FTA §12's tripwire). Reviewed at the M2 gate against FTA §12's explicit rule: definitions hold declarative metadata only; anything else composes the pattern by hand. A hand-written page is an acceptable outcome, and reviewers are told so here so they do not treat it as a failure to be abstracted away.
Hardening gets compressed The date slips, M8 becomes "the last week," and accessibility and E2E are dropped — on a tool whose users are keyboard-heavy operators moving money. M8 is a milestone with exit criteria, not a buffer. The rules it audits (formatters, focus, boundaries, error handling) are enforced continuously from M0, so M8 is a sweep rather than a rescue — which is what makes it survivable if it is compressed.

Acceptance gates

A gate is a review that a milestone must pass before the next one starts. Its value comes entirely from being refusable — a gate that has never blocked anything is a ceremony, and everyone learns to treat it as one.

The standing gate — applied at every milestone

CheckPasses when…Source
Architecture conformance Boundary lint passes with zero suppressions — or each suppression names an ADR. Axios appears only in the HTTP client. No query-key literals outside a factory. No permission-string literals outside the registry. No raw localStorage access outside the session module. FTA §4, §6, §7, §8, §14
Design System conformance No hardcoded colors, spacing, or radii — tokens only. No inline formatting: every money, date, phone, and identifier renders through the formatter module. Statuses render through the badge vocabulary, never by string comparison in a component. Design System §3, §5, §17, §27
Correctness of money paths Every mutation in the milestone: no auto-retry, no optimistic update, confirmation renders from fresh data, invalidation-map entry exists and has a test. FTA §8, §11, D-3, D-7

Who gates, and the one rule that keeps gates honest

A gate is passed by a reviewer who did not build the milestone. Self-certification defeats the purpose — not because engineers are dishonest, but because the person who built it is the person least able to see what they assumed.

A failed gate stops the next milestone; it does not "add a follow-up ticket." The follow-up ticket is how every one of these rules dies: it converts a blocking defect into a backlog item that is deprioritized against feature work forever. If a gate is worth defining, it is worth being blocked by — and if the team is unwilling to be blocked by it, it should be deleted from this document rather than quietly ignored, because a gate everyone routes around teaches the team that the rules here are optional.

09

Definition of milestone completion

"Done" is the most abused word in software planning, because everyone means something different by it and nobody says which. Here it means one thing, and it is checkable by someone who was not in the room.

A milestone is done when all seven hold — not six

(1) Functionally complete. Every deliverable listed in §3 exists and works against the real backend — not against MSW. A feature that has only ever run against a mock is a feature that has never run.

(2) Exit criteria demonstrated. Each of the milestone's exit criteria has been shown — performed in front of the reviewer — not asserted in a status update. The criteria in §3 are written to be demonstrable precisely so this is possible.

(3) The standing gate passes (§8), reviewed by someone who did not build it.

(4) Tests exist at the right level: mappers and formatters unit-tested; principal flows covered by integration tests; irreversible flows covered by E2E from M5 onward.

(5) Nothing is left dark by accident. Every feature flag introduced in the milestone has an owner, a reason, a removal condition, and a documented per-environment default (FTA §14). A flag hiding an unfinished feature is fine; a flag nobody can explain is not.

(6) Decisions are recorded. Any deviation from the four approved documents has an ADR. Any contradiction discovered in them has been raised, not worked around.

(7) No P1 defects. For this product a P1 is defined narrowly and unambiguously: anything that could cause an operator to act on a wrong financial value, or to take an irreversible action they did not intend. A cosmetic misalignment is not a P1. A stale amount in a confirmation dialog is, and it blocks the milestone.

What "done" explicitly does not mean

It does not mean "merged." Code on the main branch that has never run against the real backend is not done; it is a hypothesis with good syntax.

It does not mean "demoed." A demo shows the happy path. Done includes the 403, the 409, the dropped connection, and the empty state — which is why they are exit criteria rather than nice-to-haves, and why the Design System spent four sections on them.

It does not mean "we'll harden it in M8." M8 audits what the rules already required continuously. It is not a debt-repayment window, and treating it as one is how it becomes one.

The resource recipe — the repeatable procedure after M2

From M3 onward, adding a resource should be a procedure, not a design exercise. If it is not — if a developer has to make an architectural decision to add a resource — then either the patterns are wrong or the resource is genuinely exceptional, and both are worth noticing rather than absorbing silently. The steps below are the checklist; each cites the document that governs it, and none of them is a decision.

StepDoGoverned by
1Add the resource's permission strings to the central registry.FTA §6
2Write the API module: endpoints, DTO types (private), and the mapper — including the status vocabulary, and preserving absent-vs-zero.FTA §7, D-6
3Write the query-key factory and the query/mutation hooks; assign the staleness tier by what the data costs when it is wrong.FTA §8
4Write the zod schema (create/edit derived from one base).FTA §10

The escape hatch, and when to use it

If a resource cannot be expressed through the recipe, do not bend the recipe. Compose the pattern's parts and write the page by hand — FTA §12's escape hatch, working exactly as designed. A hand-written page is an acceptable, non-exceptional outcome and MUST NOT be treated as a failure to abstract.

What is worth escalating: the third time the recipe fails for the same reason. Once is a special case; three times is a pattern the architecture missed, and it deserves an ADR rather than a third workaround.

Day one

Before any product code

Confirm O-1 is signed. Raise the two backend tickets. Land the boundary lint rule, the config validator, and the MSW harness. Open the ADR log with ADR-0001. This is M0, and it is short — but it is a gate, and G0 is refusable.

The first product commit

Is part of the walking skeleton (M1), on Villes. One developer. Not a component library, not a folder scaffold — a login, a shell, a list, and a real record created against the real backend.

Everyone else, meanwhile

Track B (§6): formatters, StatusBadge, MoneyAmount, EntityChip, the test harness. Fully specified by the Design System, dependent on nothing, and needed by every screen that follows.

The one condition under which the frozen documents reopen

The four approved documents are constraints, and this roadmap does not revisit them. But they were written before a single line ran against the real backend, and G1 exists precisely because reality gets a vote. If implementation discovers a genuine contradiction — the pagination envelope differs from what was assumed, a permission string does not exist, a status value was missed — the document is corrected, through review, and this roadmap adjusts. That is not a failure of the specifications; it is the process working, and it is exactly what the FTA's own ADR rule (§19) was written to handle.

What must not happen is the code quietly diverging from the documents while both are believed to agree. That is how a codebase becomes undocumented while appearing documented — and it is the specific decay every gate in §8 exists to prevent.
