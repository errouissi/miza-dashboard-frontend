The legacy dashboard is a shell. The backend is the real system.

Full audit of C:\Miza\frontend against the current C:\Miza\backend API surface. Nine routes exist; five render a single placeholder line. The parts that are built — auth, routing, layout, sidebar — are sound and worth keeping. Everything else is unclaimed territory, and the backend already has far more domains (stock, deposits, cheques, grattage, allocations, transfers) than the frontend has pages for.

01

Stack & project structure

The frontend was scaffolded correctly and never filled in. It has an opinionated, modern toolchain — nothing here needs replacing, only extending.

Framework

React 19 + TypeScript, built on Vite 7. Path alias @ → src/.

UI layer

shadcn/ui ("new-york", neutral base) + Tailwind CSS v4 via the Vite plugin — no tailwind.config file. Only 9 primitives generated so far (avatar, button, dropdown, input, separator, sheet, sidebar, skeleton, tooltip).

Routing

react-router-dom v7, BrowserRouter, all routes declared flat in one file.

Data fetching

Bare axios instance with a bearer-token interceptor. No React Query / SWR — every page will hand-roll its own loading/error/cache state unless one is introduced.

Forms

react-hook-form + zod + @hookform/resolvers, used correctly on the one form that exists (Login).

Tables & charts

No table library, no chart library. Neither has been chosen yet — first real decision Phase 8 needs to make.

State management: intentionally absent

There is no Redux/Zustand/Context store. useAuth() reads directly from localStorage on every call via authStorage.ts. This is a deliberate, reasonable choice for an app this size — it should stay this way until a concrete cross-page state need appears (e.g. shared filter state), rather than being pre-emptively added.

Sidebar visibility ≠ route protection

AppSidebar filters MENU by role client-side, but AppRoutes.tsx only wraps /dashboard in RequirePermission. A logged-in commercial or manager — roles the frontend's Role type doesn't even list — can navigate to /admins directly by URL; the page just won't be in their sidebar. Today this is harmless (the pages are empty), but it's a gap to close before real admin CRUD UI lands there. See §6.

04

Auth & permissions model

Token-based, not cookie-based — matches the backend's Sanctum bearer-token design.

Frontend

login() → POST /auth/login → token + user object stored in localStorage (keys: token, user). Axios interceptor attaches Authorization: Bearer … to every request. No refresh-token flow, no token-expiry handling — a 401 mid-session has no defined UX.

Backend

Sanctum, one unified /auth/login for three tokenable models (User/Admin, Agent, Client). Spatie roles: super-admin, admin, manager, commercial — four, all on the sanctum guard. Permission strings returned in the login payload drive every route-level permission:* middleware.

Frontend's role model is narrower than the backend's

app/permissions/menu.ts types Role as "super-admin" | "admin" only. manager and commercial exist as real, permissioned backend roles but have zero representation in the frontend menu or route model. That's fine if Phase 8 stays admin-facing only — worth confirming as a scope decision rather than an oversight (see §7, Q1 open question).

06

Technical debt

Ranked by what it costs to leave alone while Phase 8 builds on top of it.

Frontend

Only one route is permission-gated

/admins, /agents, /clients, /transactions, /profile, /security are reachable by any authenticated user by URL — the sidebar hides them, the router doesn't. Needs RequirePermission wired per-route against real backend permission strings before real CRUD UI ships behind them.

Backend

Dashboard has three dead controller methods

chartData(), recentActivities(), agentsOverview() are fully implemented in DashboardController but routes/api.php only wires index(). Cheap to fix (three route lines), but the Dashboard page can't show charts or an activity feed until it's done.

Backend

Companies & Suppliers have no API

Both are first-class entities in the stock domain (bons reference them by ID) but neither has a controller. Any "manage suppliers" or "manage companies" screen is blocked until these are built — flag this to the backend track now, since Phase 8 will hit it as soon as the Bons UI needs a supplier picker.

Frontend

Role type only knows two of four backend roles

"super-admin" | "admin" — manager and commercial are invisible to the menu system. Likely fine if this dashboard stays admin/super-admin only (agents have their own app surface per the API docs), but should be a confirmed decision, not a silent gap.

Frontend

No data-fetching or table library chosen

Every future list page (agents, clients, cheques, deposits, bons…) needs pagination, filters, loading and error states. Building each by hand nine-plus times is the single biggest multiplier of effort in this phase — see §7.

Frontend

Login page bypasses the design system

Plain <input>/<button> with Tailwind utility classes, not the shadcn primitives already installed (Input/Button exist in components/ui and are unused here). Small, but it's the first thing every user sees.

Backend

Token storage has no rotation/expiry story

Bearer tokens in localStorage with no refresh flow is standard for this app's size, but worth a conscious note: a compromised token is valid until manually revoked server-side. Not a Phase 8 blocker, just an inherited constraint.

07

Recommended new dashboard architecture

Keep the shell. It's correctly built and every backend domain fits through it without structural change. The work is additive, not corrective.

Keep as-is

Router structure, ProtectedRoute, DashboardLayout/AppSidebar/AppHeader, the axios instance + interceptor pattern, shadcn/Tailwind as the design system, RHF+zod for forms.

Add: data layer

A query/cache library (TanStack Query is the natural fit for this axios-based setup) sitting on top of one service module per backend domain, mirroring src/services/ — clients.ts, agents.ts, deposits.ts, etc. Removes the need to hand-roll loading/error/refetch on every page.

Add: one table primitive

Nearly every domain in §5 is "paginated list + filters + row actions." A single composable table (TanStack Table headless + shadcn styling) built once against the Clients list, then reused for agents, cheques, deposits, bons, etc., rather than rebuilt per page.

Add: one lifecycle-detail primitive

Bons, Allocations, Agent Transfers, and Agent Stock Returns all share the identical backend shape: draft → add/edit/remove lines → validate (irreversible). One detail-page pattern (header + editable line table + validate action with confirmation) serves all four instead of four bespoke screens.

Add: permission-driven routing

Replace the two independent gates (§3, §4) with one: derive route access directly from the permissions[] array the login response already returns, the same array RequirePermission already reads — so menu visibility and route protection can't drift apart again.

Add: charts, once routes exist

A lightweight charting library (Recharts fits a shadcn-styled dashboard well) for the deposits-over-time / by-method / agent-registration series — blocked only on wiring the three dead controller methods in §6.

Open question worth a decision before build starts

Q1 — scope of roles. Is this dashboard admin/super-admin only, or does it also need to serve managers and commercials (who already have a separate /agent/* API surface, per the backend docs)? This determines whether §4's Role type needs widening or whether the current two-role model is correct as-is.

08

Recommended implementation order

Sequenced by dependency and payoff — each step either unblocks the next or ships something visible on its own.

  1. 1

    Wire the three dead dashboard routes

    Add route lines for chartData, recentActivities, agentsOverview — backend logic already exists.

    Why first: near-zero backend effort, unblocks the highest-visibility frontend page.

  2. 2

    Build the Dashboard page

    Stat tiles, charts, recent-activity feed against the now-complete stats API.

    Why: it's the landing page every session hits; also the first place the table/chart primitives from §7 get built and proven.

  3. 3

    Clients module

    List + filters, detail, assign/reassign/bulk-assign, status toggle.

    Why: backend explicitly marked stable/complete, broadest existing filter surface — the best place to finalize the shared table pattern.

  4. 4

    Agents module (managers & commercials)

    List, detail, multi-file create form, block/activate, grattage-outstanding badge.

    Why: reuses the Clients table pattern; the create form is the first real multi-file-upload UI needed.

  5. 5

    Money domain: Cheques, Deposits, Debt Payments

    Submission, pending queue, approve/reject/annuler; deposit validate/reject.

    Why: highest business sensitivity (irreversible financial actions) — sequenced after the team has proven the patterns on lower-stakes domains.

  6. 6

    Grattage invoices (read + cancel only)

    List/detail/cancel. Explicitly skip building UI for admin-create or settle — both are hard-403'd by backend design.

    Why: depends on Agents (for outstanding-obligation context) landing first.

  7. 7

    Stock chain: Bons → Allocations → Agent Transfers → Agent Stock Returns

    Build the shared draft/lines/validate detail primitive once against Bons, then reuse for the other three in sequence.

    Why: identical backend shape across all four — one build, three reuses. Ordered by position in the physical stock chain (supplier-facing first).

  8. 8

    Admins management

    List/create/update/block/delete — super-admin only, low traffic.

    Why: narrowest audience, safe to defer without blocking anything else.

  9. 9

    Reference data: Villes, Secteurs, Products

    Simple CRUD tables, no lifecycle, no filters of consequence.

    Why: lowest complexity and lowest urgency — good backlog filler, not a dependency for anything above.

Explicitly out of scope until backend work lands

Companies & Suppliers admin UI — cannot start until controllers/routes exist for these models (§6). Recommend raising this as a small backend ticket before Bons UI (step 7) needs a supplier picker.
