Miza Frontend Technical Architecture

The engineering blueprint for building the dashboard the approved Architecture proposed, styled the way the approved Design System requires. This document decides how the frontend is organized — its layers, boundaries, data flow, and rules — so that a senior React developer can start work without making a single major architectural decision alone. It is technology-aware (it names the stack it binds to) and implementation-independent (it contains no code).

Version 1.0 — proposed Inputs Discovery ✅ · Domain Architecture ✅ · Design System v1.0 ✅ · backend through Phase 5.10 Status Awaiting review before stage 5 (implementation)

How to read this document

MUST marks a rule with no exceptions — violating it is an architecture bug and a valid reason to reject a pull request. SHOULD marks the default; deviating requires a stated reason in review. MAY marks an explicit freedom. Callouts marked ⚖ Decision record a place where a conventional or tempting choice was rejected, and why — future contributors inherit the reasoning, not just the rule. Decisions are numbered D-n and become the seed of the ADR log (§19).

Where this document references the upstream specifications it implements, it cites them as Discovery, Architecture §n, and Design System §n. Those documents are approved and binding; where this one appears to contradict them, they win and this one is wrong.

The stack this binds to — inherited, not chosen here

React 19 · TypeScript · Vite 7 · Tailwind v4 · shadcn/ui (new-york, neutral) · react-router · React Hook Form + zod · axios. Authentication is Laravel Sanctum bearer tokens; authorization is Spatie permission strings returned in the login payload. All of this already exists (Discovery §1) and is kept — the Discovery report's verdict was "the frontend was scaffolded correctly and never filled in," and this architecture is additive to that scaffold, not a replacement for it.

Two libraries are added by this document, both anticipated by Discovery §7: TanStack Query as the server-state layer (§8) and TanStack Table (headless) beneath the DataTable (§12). They are the only dependencies introduced by this architecture stage — which is a statement about what this document decides, not a permanent freeze on the dependency list.

The dependency policy, stated once and applying to everything after this document: a new dependency is admissible when a concrete product requirement cannot be served safely or reasonably by the existing stack — and not otherwise. Convenience, familiarity, and "it's only a few kilobytes" are not requirements. Every meaningful dependency MUST be justified through architectural review and recorded as an ADR (§19), stating what it does that the current stack cannot. Prefer small, focused, actively maintained libraries with a narrow surface. Reject dependencies that duplicate a capability already present, or that impose a second architectural model competing with the one specified here — a second data-fetching layer, a second form system, or a second styling paradigm are all rejections by default, because the cost is not the bundle, it is the fork in how the codebase thinks.

01

Frontend architecture philosophy

The dashboard moves real money and real stock through irreversible steps. That single fact — not fashion, not framework convention — is what every principle below is derived from. An architecture for a content site optimizes for time-to-first-render; this one optimizes for never showing an operator a number that is wrong, and for still being modifiable in three years by someone who wasn't here.

The backend's domain model is the frontend's domain model

The five domains (Network, Money, Stock, Grattage, Reference) are not a UI grouping — they are boundaries the backend genuinely enforces, with Grattage as the one deliberate seam between the Stock and Money ledgers (Architecture §2). The frontend mirrors them exactly: folders, routes, query keys, and permissions all use the same names.

Why: when the frontend invents its own decomposition, every backend change requires a translation step that lives only in a developer's head. Shared vocabulary makes "where does this go?" answerable without a meeting.

Correctness over convenience, explicitly

Where a convenient pattern trades a small amount of correctness for a large amount of developer comfort, this architecture takes the correctness. That is why there are no optimistic updates on money (§8), why mutations never auto-retry (§11), and why data that gates an irreversible action MUST be refetched before that action is offered (§8).

Why: the cost of a wrong render here is not a bad review — it is a cheque approved twice or a deposit settled against the wrong invoice. Optimism is a UX technique that assumes failure is cheap; in this product it isn't.

One shape per repeating problem

Inherited directly from Architecture §1. The backend repeats two shapes relentlessly — "paginated list with filters" and "draft → add lines → irreversible validate." The frontend builds each shape once and configures it per domain (§12). Fifteen near-identical CRUD trios hand-built is the single biggest waste available to this project, and the Discovery report named it as such.

Why: a bug fixed in a shared pattern is fixed everywhere; a bug fixed in one of fifteen copies is fixed once and survives fourteen times.

Separation of concerns, enforced not encouraged

Four layers (§2) with a strict, one-directional dependency rule (§4). "Enforced" means a linter fails the build on a forbidden import — not a convention in a README that erodes in the third sprint under deadline.

Why: every architecture document ever written has been ignored under pressure. The only boundaries that survive are the ones a machine checks. If a rule in this document cannot be mechanically enforced, it is written as a review checklist item and we accept that it is weaker.

Scalability means the sixth domain is boring

The measure of this architecture is not how well it renders today's five domains — it is how much work a domain that does not exist yet will cost. The target: a new resource ships as a folder plus registry entries (a definition, a query-key factory, a route contribution, permission strings), with no change to any shared file.

Why: the business is adding domains (Phase 5.10 added reconciliation; more will come). An architecture that requires touching shared code for every addition creates a merge-conflict bottleneck exactly when the team grows.

Maintainability means deletable

Every feature MUST be removable by deleting its folder and its registry entries. If deleting a domain leaves dangling imports across the app, the boundaries have already failed. Cross-domain coupling is permitted only through a domain's declared public surface (§4), never into its internals.

Why: the ability to delete is the only honest test of whether a boundary exists. It is also what makes the retired-route problem (the hard-403'd grattage endpoints) survivable rather than a slow leak of dead code.

D-1 · The frontend is a client of a domain-rich backend, not a place to re-implement the domain

Tempting alternative: build a rich client-side domain model that mirrors backend business rules — capacity formulas, restock-gate logic, settlement arithmetic — so the UI can validate everything instantly without a round trip. Rejected.

Duplicated business logic is a guarantee of eventual divergence: the day the backend changes a capacity formula and the frontend does not, the UI confidently shows an operator a number the server will refuse to honor. The backend is the sole authority on domain truth (it owns the transactions and the invariants), and the frontend's job is to ask it, present the answer faithfully, and explain refusals well (§11's error-code registry). Client-side rules are permitted only as UX affordances that mirror a server rule — required fields, formats, min/max — and the server remains the enforcer of every one of them. Where the frontend needs a derived business value (an agent's capacity, a restock-gate state), it fetches it; it does not compute it.

02

High-level architecture

Four layers. Dependencies point downward only — every arrow in the diagram below is one-way, and there are no exceptions granted anywhere in this document.

App layer composition root · knows everything, is known by nothing

The only place where the application is wired: the router and its route table, the provider stack (query client, session, error boundary, toaster), the shell layouts (AppShell, and the five page skeletons' outermost frames), and application bootstrap (config validation, session hydration).

Owns: what exists and how it is assembled. Never contains: business logic, API calls, or domain types. If a file in app/ knows what a cheque is, it is in the wrong layer. The app layer is the one place allowed to import from every domain, because composition is its entire job.

↓ imports

Domain layer the product · where features actually live

One folder per business domain (Network, Money, Stock, Grattage, Reference, Overview), each containing its resources (cheques, deposits, bons, …). A resource owns its API module, its DTO mappers, its domain types, its query keys and hooks, its pages, and its domain-specific components.

Owns: everything specific to one business concept. Rule: a domain MUST NOT import another domain's internals — only its declared public surface, and only where a real backend coupling justifies it (§4). Most domains import no other domain at all.

↓ imports

Shared layer the product's vocabulary · domain-agnostic, product-specific

Things every domain uses but no domain owns: the page patterns (ListPage, DetailPage, DraftLifecyclePage, WorkspacePage, ApprovalQueuePage, CreateWizard), the business components (StatusBadge, MoneyAmount, EntityLink, DataTable, FilterBar, LineItemsEditor, PermissionGate…), the shadcn primitives, the formatters that implement Design System §5/§27, and cross-cutting hooks.

Rule: the shared layer MUST NOT import from any domain. A shared component that needs to know what a cheque is has been mis-placed — it is a domain component, or it needs its cheque-ness passed in as configuration. This is the boundary that decays first and matters most.

↓ imports

Infrastructure layer the plumbing · knows nothing about Miza

The HTTP client and its interceptors, error normalization, the query client and its defaults, the session/token store (the only module that touches localStorage), the permission registry and evaluator, configuration and feature flags, the logger and error reporter.

Rule: infrastructure MUST NOT import from shared or domain layers. It is theoretically liftable into another product with the domain names removed — and that is exactly the test to apply when deciding whether something belongs here. The permission registry's contents are Miza-specific data; the permission evaluator is not, and they are separated accordingly (§6).

The dependency rule, stated once

MUST: app → domains → shared → infrastructure. Never sideways (domain → domain, except through a public surface), never upward (shared → domain, infrastructure → anything). Enforced by an import-boundary lint rule that fails CI, not by good intentions (§4).

The single most common violation to watch for: a shared component that imports a domain type "just for the props." That is the crack through which the shared layer becomes a second, worse copy of the domain layer. Pass the data in; do not reach for it.

D-2 · Feature-first, not layer-first — and explicitly not Atomic Design

Two conventional alternatives were considered and rejected.

Layer-first (/components, /hooks, /services, /types at the root, all domains mixed inside each) is what the legacy build started doing. It fails at exactly this scale: a change to cheques touches four distant folders, nothing tells you which files belong together, and deleting a feature becomes an archaeology exercise. It optimizes for "where do I put a hook?" — a question asked once — over "what makes up this feature?" — asked every day.

Atomic Design (atoms/molecules/organisms) is rejected outright. It sorts components by visual complexity, a property with no relationship to ownership, change frequency, or deletability. In practice every team argues about whether something is a molecule or an organism, and the taxonomy answers no question anyone actually has. The shared layer's three folders (ui / patterns / business) sort by ownership and origin instead — generated, skeleton, or authored-business — which are the distinctions that determine who may edit a file and when.

Two folder rules that prevent the usual decay

(1) No utils/ folder. "Utils" is where code goes to escape classification, and it grows without bound because nothing is ever wrong to put there. Helpers live with the thing they serve; genuinely universal ones are named for what they do (shared/formatters), not for what they aren't.

(2) One barrel per resource, and no deeper. Each resource exposes exactly one index.ts — its public surface (§4). There are no barrel files inside a resource, and none in shared subfolders: deep barrel chains create circular imports that are excruciating to debug and defeat tree-shaking, and they hide exactly the dependency edges this architecture is trying to make visible.

04

Feature organization & boundaries

Boundaries that are not enforced are not boundaries. This section defines what may import what, and how the build fails when someone doesn't.

The public surface rule

Every resource has exactly one front door

A resource's index.ts declares what the rest of the app may use: typically its route contributions, its query-key factory, a small number of hooks, its domain types, and any component genuinely meant for outside consumption. Everything else — the API module, the mappers, the page components, the internal state — is private.

Reaching past the front door is a build failure

Importing domains/money/cheques/api/… from outside the cheques folder is forbidden and lint-enforced. The distinction matters because the front door is a contract: internals may be refactored freely, and the only way that stays true is if nobody depends on them.

Cross-domain logic — the three sanctioned crossings

Architecture §6 documents six real cross-module workflows. They resolve into exactly three mechanisms, and every crossing in the app MUST use one of them. Anything else is a boundary violation wearing a business justification.

MechanismHow it worksUsed by
1 · Composition at the page A page imports the public surface of several domains and arranges their panels. The composing page owns the layout; each panel remains owned by its domain. No domain learns about another. Agent 360 and Client 360 (WorkspacePage) — the only layout that crosses domains by design (Architecture §4).
2 · Public hook import One domain imports a hook from another domain's public surface because the backend genuinely couples them. The importing domain gets a value; it does not get the other domain's internals or its business rules. The restock gate. A Grattage rule that Stock's Allocation and Agent-Transfer forms must respect (Architecture §5). Grattage exports the gate hook; Stock consumes it. This is the one intentional domain→domain import in the app, and it is documented here precisely so it stays the only one.
3 · The invalidation map A mutation in one domain must refresh data in another. The mutation does not import the other domain's query keys; it emits a named domain event, and a single central map declares which keys that event invalidates (§8). Cheque approval → agent capacity (Money → Stock). Deposit validation → grattage invoice settlement (Money → Grattage). Any mutation whose effects the backend propagates across domains.

D-3 · Cross-domain cache invalidation goes through a central map, not through direct key imports

The obvious approach: when the cheque-approval mutation succeeds, import Stock's query keys and invalidate the agent-capacity query directly. It is three lines and it works. Rejected.

It works today, and it quietly makes Money depend on Stock, then on Grattage, then on Overview — because every mutation with cross-domain effects adds another import. Within a year the domain graph is fully connected and the boundaries in §2 are decorative. Worse, the coupling is invisible: nothing lists which mutations affect which caches, so a developer adding a sixth domain has no way to discover that approving a cheque should now also refresh their widget.

The invalidation map inverts it. Money's mutation announces cheque.approved and knows nothing else. One file in infrastructure/query/ maps that event to the affected key prefixes across every domain. The coupling still exists — it is a real business coupling and cannot be wished away — but it is now declared in one readable place instead of smeared across mutation callbacks. The cost is one indirection; the benefit is that "what happens when a cheque is approved?" has a single, greppable answer.

Enforcement — how these rules actually hold

The import matrix MUST be encoded as an ESLint boundary rule (import/no-restricted-paths or an equivalent boundaries plugin) and MUST fail CI, not merely warn. A warning is a rule with an opt-out, and under deadline every opt-out is taken.

The escape hatch is deliberate and expensive: a violation may be merged only with an inline suppression comment that names an ADR (§19). That makes exceptions possible — occasionally they are genuinely right — while guaranteeing that each one is visible, argued, and countable. If the suppression count grows, the architecture is wrong and should be amended rather than bypassed.

05

Routing architecture

Routes are contributed by domains and assembled by the app layer. No central file lists every screen by hand — that file is always out of date, and the Discovery report already found five legacy routes with no permission guard at all because of exactly that pattern.

Route ownership

Domains own their routes

Each resource declares its own route contributions in routes.ts: path segment, the lazy-loaded page component, the permission string required, and the breadcrumb label. Adding a screen means editing one file inside one domain.

The app layer assembles them

app/router/ collects every domain's contributions into the route tree and wraps them uniformly — guard, layout, error boundary, suspense. Uniformly means every route, with no per-route hand-wrapping to forget.

Guards are generated, never hand-written

Because the permission string travels with the route declaration, a route that exists is a route that is guarded. Forgetting the guard is not possible — there is no unguarded code path to forget it in (§6).

Route structure

/overview The landing page. Role-aware widget grid; owns no CRUD (Architecture §5).
/{domain}/{resource} ListPage — e.g. /money/cheques, /stock/bons. Filter, sort, and page state live in the query string (§9), so the URL is the view.

D-4 · Lazy-load per domain, not per route — and no route-level data loaders

Code splitting: the natural instinct is to lazy-load every page component. Rejected in favor of splitting at the domain boundary. An operator working in Money will visit the cheque list, a cheque detail, and the pending queue within seconds of each other; splitting those three into three chunks buys a few kilobytes on first paint and then charges a network round trip at every navigation — on an office connection this reads as lag exactly where the product should feel instant. One chunk per domain, plus a shared chunk for the pattern library, matches how people actually move through the app (§15).

Data loading: react-router's loader/action APIs are deliberately not used. Data fetching belongs to TanStack Query (§8), and having two systems that both fetch — one keyed by route, one keyed by query key, with two caches and two invalidation stories — is a well-documented way to produce data that is fresh in one and stale in the other. The router routes; Query fetches. One responsibility each.

Navigation philosophy

The URL is the state (§9). Any view an operator can reach by clicking MUST be reachable by pasting a link — filters, sort, page, and active tab included. Design System §15 makes this binding: "every filtered view is shareable and bookmarkable from day one." A filtered table whose state lives in React state instead of the query string breaks that contract silently, and the bug only surfaces when someone pastes a link into a chat and their colleague sees different rows.

Navigation is never a substitute for a drawer. Design System §18 already decided this: modal = a decision, drawer = a task, page = a workspace. Routing follows that, not the other way round — we do not invent a route because it is architecturally convenient to have one. A drawer's open state MAY be reflected in the URL where the drawer is deep-linkable (a record preview); a drawer that is a transient sub-task of the current page MAY keep its state local.

06

Permission architecture

One registry, four consumers. The Discovery report found the legacy build's sidebar and route guards drifting apart — items visible that led nowhere, routes reachable that should not have been. That class of bug is not fixed by being careful; it is fixed by making the two read from the same source.

D-5 · Authorize on permission strings, never on roles

The tempting shortcut is everywhere in dashboards: if (user.role === 'admin'). It reads well, it requires no registry, and it is wrong. Rejected without exception.

Roles are a grouping of permissions and they change: the backend has four (super-admin, admin, manager, commercial) and the Architecture explicitly anticipates managers and commercials gaining dashboard access later. Every role check written today is a line that must be found and re-reasoned on that day — and they cannot be found reliably, because they are scattered across components as inline conditionals. Permission strings, by contrast, are what the backend's permission:* middleware actually enforces; checking the same string the server checks means the UI and the API can never disagree about what is allowed.

The one legitimate use of a role: presentation that genuinely varies by who someone is (a manager's Overview showing different widgets than an admin's). Even there, prefer a permission if one exists. Roles describe people; permissions describe actions — and the UI gates actions.

The registry and its consumers

PieceResponsibility
Registry data A single map of domain → action → backend permission string, mirroring the backend's own App\Authorization\*Permissions classes so both sides use one vocabulary (Architecture §10). It is data, and it is the only place a permission string literal appears in the codebase. A permission string typed inline in a component is a bug, because it cannot be renamed, audited, or found.
Evaluator logic Answers "does the current session hold permission X?" against the permission list the backend returned at login (Discovery §4). Pure, synchronous, trivially testable. Lives in infrastructure and knows nothing about Miza's domains — it evaluates strings.
1 · Route guards Generated from each route's declared permission (§5). Missing permission → the calm 403 state of Design System §23, not a redirect to Overview: silently bouncing someone somewhere else is how users conclude the app is broken.
2 · Sidebar A nav group renders only if the user holds at least one permission in it; a sub-item renders only with its own (Architecture §3). Same registry as the guards — one list, two consumers, no drift. This is the specific fix for the Discovery finding.
3 · UI guards The PermissionGate component wraps any action: absent permission means the control is not rendered. Design System §10 is explicit that a disabled control must explain itself; "you lack the permission" is not an explanation an operator can act on, so unavailable-by-permission means absent, while unavailable-by-state (a validated bon cannot be re-validated) means disabled-with-a-reason.
4 · Action guards The last line: a mutation hook refuses to fire without its permission. This is belt-and-braces against a button rendered by mistake, and it keeps the check next to the dangerous thing rather than only next to the pretty thing.

Client-side permissions are UX, never security

MUST be understood by everyone who touches this code: every gate in this section is a convenience — it stops an operator from seeing a button that would fail. It stops nobody from calling the API. The permission list lives in localStorage and can be edited by anyone with a browser console.

The backend is the only authority (§17). It is why a hidden button is not a security control, why the frontend never "trusts" a permission for anything but rendering, and why a 403 from the API must always render gracefully rather than being treated as impossible — because the gate can be bypassed, and when it is, the API is what holds.

Future extensibility — what a new role costs

Nothing structural. Because no component branches on a role, and because routes and nav items carry permission strings declaratively, granting managers dashboard access is: (1) the backend adds permissions to the role; (2) the login payload carries them; (3) the sidebar and routes light up. Zero frontend changes to enable a role that uses existing permissions. A genuinely new capability adds one registry entry and the route that declares it. That is the whole cost, and it is the design goal.

07

API layer

Three concentric rings: one HTTP client (infrastructure), one API module per resource (domain), and a mapping seam between the backend's wire format and the frontend's domain model. The mapping seam is the part that will feel like overhead for the first week and save the project thereafter — §7's central argument.

The HTTP client — one instance, four responsibilities

What it does

(1) Attaches the bearer token from the session store. (2) Sets the base URL from validated config. (3) Normalizes every failure into a single AppError shape (§11) — network failures, 4xx, 5xx, and timeouts all arrive at calling code looking the same. (4) Handles 401 centrally by terminating the session (§17) — closing the gap Discovery flagged, where "a 401 mid-session has no defined UX."

What it must not do

It MUST NOT know about domains, retry mutations (§11), show UI, or be imported outside a resource's api/ folder. MUST: axios appears in exactly one infrastructure module and is otherwise invisible — a component that imports axios is a bug. That single rule is what makes swapping the transport (or adding request signing, or tracing) a one-file change.

D-6 · A real mapping layer (anti-corruption layer), not "just use the API response"

The pragmatic position — "the backend returns JSON, type it, render it, why add a translation step nobody asked for?" — is the majority practice and it is rejected here, on evidence from this specific backend rather than on principle.

The evidence: Design System §17 documents that the backend's status vocabulary is genuinely mixed — English in some domains (draft, settled, validated) and French in others (en_attente, accepter, rejetee, annuler) — and it rules that the badge system "maps by meaning, never by string." That mapping must happen somewhere. Without a mapper it happens in components, and then status === 'accepter' || status === 'validated' is copy-pasted into every table, badge, and filter in the app — thirty places to fix when the backend normalizes its vocabulary, which it eventually will.

With a mapper, the string chaos stops at the edge of the domain, exactly as the Design System requires ("the string chaos stays in the API layer, never on screen"). The frontend's Cheque has a semantic status; the DTO's raw string is an implementation detail of the transport. The cost is one small file per resource. The benefit is that a backend rename becomes a one-line change, and that the domain model is free to be shaped for the UI's needs (money as a typed value, dates parsed once, entities normalized per Design System §29) rather than for the ORM's.

Scope discipline: mappers translate shape and vocabulary. They do not compute business values (D-1). A mapper that starts calculating capacity has become a domain model and must be stopped in review.

DTOs, domain models, and response normalization

DTO types mirror the wire format exactly, including its inconsistencies, and are private to the resource — never exported past index.ts, never used as a component prop. If a DTO type escapes into a component, the mapping layer has been bypassed and the boundary is already leaking.

Pagination is normalized once, in shared types: Laravel's envelope becomes one Paginated<T> shape used by every list in the app, so the DataTable consumes one contract rather than each domain's flavor of it. The per-page ceiling (per_page ≤ 100, Design System §14) is a client-side constant that mirrors the API clamp — surfaced to the user in the page-size selector, never discovered by getting clamped.

Absent vs zero (Design System §5) is preserved through the mapper: a null amount stays null and renders as —. A mapper that coalesces null to 0 for convenience destroys a financial distinction and is a defect, not a tidy-up.

08

TanStack Query strategy

Query is the server-state layer, and — stated once, because it is the rule most often broken — it is also the cache. Server data is never copied into component state, never mirrored into a store, never "kept in sync." There is one copy, and Query owns it.

Query keys

Structure

Hierarchical, most-general to most-specific: [domain, resource, scope, params] — e.g. ['money','cheques','list',{status:'pending',page:1}] or ['money','cheques','detail',id].

Why hierarchical: invalidation is prefix-based. Invalidating ['money','cheques'] refreshes every cheque view at once; invalidating ['money','cheques','list'] leaves open detail pages untouched. Flat or string-concatenated keys make that impossible and force manual bookkeeping of what to refresh.

Factories, not literals

MUST: every key comes from its resource's key factory. A key array typed inline in a component is forbidden — it is unfindable, un-renameable, and one typo away from a cache that silently never invalidates (the worst failure mode available here, because it looks like it works).

The factory is part of the resource's public surface (§4): it is what the invalidation map references, and what other domains use when they legitimately need to read across a boundary.

Caching philosophy — staleness is a business decision, not a default

A single global staleTime is the standard advice and it is wrong for this product, because the cost of stale data here ranges from zero (a ville's name) to a wrongly-approved cheque. Tiers are assigned per data class, and each tier's number is derived from what that data costs when it is wrong.

TierDataStalenessRationale
Static long Reference: villes, secteurs, products. Hours. Refetch on window focus: off. Changes are rare and administrative. These feed every form's pickers, so caching them hard removes a request from nearly every screen. A ville renamed an hour late costs nothing.
Slow minutes Network identity: agent and client lists and details. A few minutes. Changes during a working day but not during a task. Stale identity data is a cosmetic annoyance, not a financial error.
Live seconds Financial queues and balances: pending cheques, pending deposits, agent capacity, outstanding grattage, restock-gate state, Overview KPIs. Seconds. Refetch on window focus: on. Multiple admins work these queues simultaneously. Showing a cheque as pending after a colleague approved it produces duplicate work and a confusing 409. Focus-refetch is the cheapest possible fix for "I alt-tabbed back after lunch."
Critical zero Anything read in order to decide an irreversible action: the record inside a validate/approve confirmation, the capacity a submission will consume, the gate state a form depends on. Always fresh — see the rule below. This is the tier that justifies the whole scheme.

The freshness rule — the most important rule in this section

MUST: data that gates an irreversible action is refetched immediately before that action is confirmed, and the confirmation dialog (Design System §19) renders from that fresh read — not from whatever was in the cache when the operator opened the page ten minutes ago.

Design System §19 requires the confirmation body to state the specific consequence with named entities and amounts, because "confirming which cheque for how much is the entire point." A confirmation dialog rendered from stale cache is a lie told at the exact moment the operator is trusting us most. If the refetch reveals the record changed underneath them (someone else approved it; the amount was corrected), the dialog MUST say so and refuse the action rather than submitting against a version the operator never saw.

Mutations, invalidation, and the optimism question

D-7 · No optimistic updates in v1 — none, not "sparingly"

Optimistic updates are the reflexive best practice: apply the change locally, roll back on failure, the UI feels instant. Rejected wholesale for this product, and it is worth being precise about why, because "we'll just use them carefully" is how this decision usually dies.

Optimism trades correctness for perceived speed, and it is a good trade when failure is cheap and rollback is invisible. Here, every mutation worth optimizing is a mutation whose failure is expensive and confusing: approving a cheque, validating a deposit that settles a grattage obligation, validating a stock movement that debits one agent and credits another. An optimistic rollback on one of those shows an operator a cheque that was approved, then wasn't — and there is no good copy for that. It is worse than a spinner, and the whole point of Design System §11's submission rule is that the form suspends anyway, so the perceived-speed gain is small.

The honest counter-argument: what about trivially reversible UI state (a favourite toggle, a collapse)? There isn't any in this product — the mutations are all financial. If a genuinely trivial one appears, it earns an ADR, not a blanket exemption granted in advance.

Instead: mutations show their pending state on the triggering control (Design System §10), the form suspends, and on success the affected queries invalidate. Fast, honest, and there is no state in which the screen shows something the server has not confirmed.

ConcernRule
Invalidation On success, a mutation invalidates by prefix — typically its own resource's list and the affected detail. Cross-domain effects go through the invalidation map (§4, D-3), never through a direct import of another domain's keys. Prefer invalidation over manual cache writes: hand-patching the cache re-implements the server's business logic on the client (D-1) and drifts the moment the backend does something the patch didn't anticipate — such as the settlement cascade a deposit validation triggers across grattage invoices.
Mutation retries MUST NOT auto-retry. Ever. A retried deposit validation is a plausible double-settlement, and a retried approve is a plausible double-approve; whether the backend happens to be idempotent on a given endpoint is not a property the frontend may assume. Failed mutations surface an error with an explicit, human-initiated retry (Design System §23). Queries, which are safe by nature, retry — see §11.

(2) A detail record on row hover/focus in a list, where the operator has all but told us where they are going. Prefetching is a courtesy, never a correctness mechanism: nothing may depend on a prefetch having happened.
Infinite queries Almost never. The one legitimate use is the entity-picker autocomplete (Design System §12), where a search returns a long candidate list and a "load more" is genuinely right. Introducing an infinite query anywhere else requires an ADR that explains why the Design System's prohibition does not apply.

When NOT to use Query

Query is for server state: data the frontend does not own and cannot recompute. It is the wrong tool for, and MUST NOT be used for: form state (that is RHF, §10); UI state (local useState); filter/sort/page state (the URL, §9); derived values (compute at render, §9); the auth token (the session store, §17); and one-shot imperative actions with no cacheable result — triggering a file download, say — which are plain function calls, not mutations pretending to be one.

The tell that Query is being misused: a useEffect that copies query data into useState. That code has just created a second, staler copy of the truth, and every bug it causes will look like a caching bug rather than the state-duplication bug it is.

State management

Most "state management problems" are classification problems: state was put somewhere that cannot own it, and the bugs that follow are blamed on the tool. This section classifies every kind of state this app has and assigns each exactly one home. Getting this table right is worth more than any library.

KindDefinitionHomeWhy there
Server state Anything the backend owns: records, lists, capacities, statuses. TanStack Query (§8). One copy, never mirrored. It is not ours. It goes stale on its own, it can change without us, and it needs caching, deduplication, and refetching — which is precisely and only what Query does.
URL state Filters, search query, sort column and direction, page, page size, active tab, deep-linked drawer. The query string. Read and written through one typed hook (§13). Design System §15 makes it binding: filter state is "one serializable, URL-encodable value" and every filtered view is shareable and bookmarkable. Lifting it to a global store is the classic over-engineering move: it makes every keystroke a global event and buys nothing, because nothing outside the form needs to know what someone half-typed.
Client/UI state Is this drawer open, which row is expanded, which rows are selected for a bulk action. Local useState, at the lowest component that needs it. It is ephemeral and scoped. Row selection in particular MUST stay local: it is meaningless outside the table, and it must be cleared when the filters change — a globally-stored selection that survives a filter change is exactly how someone bulk-assigns 100 rows they can no longer see (Design System §14).

D-8 · No global state library — not Redux, not Zustand, not yet

The Discovery report found no store and called its absence "a deliberate, reasonable choice for an app this size." This document confirms that and states the reasoning, because the pressure to add one arrives the first week and is always framed as inevitable.

Look at the table above: once server state lives in Query, URL state lives in the URL, form state lives in RHF, and session lives in one context, the residue that a global store would hold is nothing. Adding Redux or Zustand to this app would create an empty container that developers then feel obliged to fill — and what gets put in it, invariably, is a copy of server data, which immediately becomes a second source of truth that goes stale and produces exactly the bugs the store was supposed to prevent. The store does not solve a state problem; it creates one.

The trigger to revisit — stated so this is a decision and not an ideology: a genuine piece of cross-page, client-owned, non-URL state that outlives a route change. A multi-page bulk-operation basket would qualify. If one appears, add the smallest thing that solves it (a context, or Zustand), via ADR — not preemptively, and not "while we're in there."

What MUST never be stored globally

Server data (it belongs to Query; a global copy is a stale copy). Permissions, beyond the single session context — re-deriving or re-storing them per feature is how the drift the Discovery report found gets reintroduced. Filter state (it belongs in the URL, or shareable links break). Form values (they belong to the form). Modal and drawer open-flags (they belong to the component that owns the modal; a global isModalOpen registry is a well-travelled road to two modals open at once, which Design System §8 calls a design error).

The unifying principle: state belongs at the narrowest scope that can own it. Every widening of scope is a new way for two parts of the app to disagree.

Context usage

React context is for values that are genuinely global and rarely change. It is not a state manager, and a context whose value changes frequently re-renders every consumer — which is why the sanctioned list is short and closed:

SessionContext

User, permissions, auth actions. Changes at login/logout only.

QueryClientProvider

Infrastructure, provided once at the root. Not application state.

Toast/confirm portals

The imperative escape hatches for notifications (Design System §20) and confirmation dialogs (§19) — one instance, app-wide, so a confirm can be requested from anywhere without prop-drilling a dialog through six layers.

Adding a fourth context requires justifying why the value is global, why it changes rarely, and why prop-passing or composition does not serve. "It was getting annoying to pass down" is not that justification — it is usually a sign the component tree is wrong.

Form architecture

Forms are where this product accepts the values that become money. The architecture here is deliberately conservative: one library, one schema per resource, one submission path, and the server as the final authority on every rule.

Validation — the schema is the contract

One zod schema per resource operation

The schema is the single source of truth for a form's shape and rules, and it is the resource's property (it lives in model/). Create and edit variants are derived from one base schema rather than written twice — two hand-maintained schemas for one resource is two places for a rule to be right in one and wrong in the other.

Types flow from the schema, not alongside it. A schema and a manually-written type that can disagree will eventually disagree.

Client rules mirror server rules — and never replace them

Every client-side rule exists to give fast feedback on a rule the backend already enforces (required, format, min/max, ≤ current debt). Design System §13 requires constraints be stated before failure, in helper text — so the schema and the helper text are written together, from the same backend rule.

MUST NOT encode a business rule the backend does not enforce. That is D-1's prohibition in its most tempting form: a frontend-only rule is an invariant with no enforcement, and it will be bypassed by the first API client that isn't this app.

Form composition & reusable patterns

A field owns its label, control, and the reserved helper/error line — as one unit, because Design System §13's layout-shift rule requires that line to be reserved space. If the reserved line is a per-form responsibility, it will be forgotten in one form and the form will jump on first error. Building it into the field primitive makes the rule structural rather than remembered.

D-9 · The wizard is multi-step React Hook Form, not a state machine library

A wizard with conditional steps is the textbook argument for XState, and the textbook argument is rejected here. There is exactly one wizard in the entire application, its steps are near-linear, and its only real complexity is a role-conditional branch (manager vs commercial fields) and a set of file uploads.

Introducing a state-machine library for one flow means every developer must learn it to touch that flow, forever, and it invites the next person to model things as machines that are better as functions. A single RHF form with per-step validation, a step index, and a review step handles this completely — and keeps the entire onboarding flow readable by anyone who knows the stack they were hired for.

The revisit trigger: a second or third wizard with genuinely non-linear, condition-heavy step graphs. At that point the abstraction earns its keep and gets an ADR. One wizard does not.

The submission flow — one path, no variations

(1) Validate the full form on submit; focus moves to the first invalid field (Design System §13). (2) Suspend: the triggering button enters its loading state and the form is disabled — Design System §11 requires double-submission of a deposit validation to be "structurally impossible, not merely unlikely," and this is where that is delivered. (3) For irreversible actions, confirm against freshly-fetched data (§8's freshness rule) with the consequence stated in the domain's own words. (4) Mutate — never optimistically (D-7), never auto-retried (§8). (5) On success: invalidate, toast, and navigate or close per the flow. (6) On failure: the form stays open with every value intact, errors mapped to fields, and an explicit retry available (§11).

Step 6 is not a nicety. Design System §23: "A lost connection MUST NOT cost a filled onboarding wizard its data." A form that clears itself on a network error has converted a transient blip into twenty minutes of re-typing, and it will happen on the longest form in the app.

Server validation errors are the same errors

A 422 from Laravel maps field-by-field onto the form using the identical rendering path as a local zod failure (Design System §13: "one visual system, two sources"). This works only if field names survive the mapping layer — which is one more reason the mapper (§7, D-6) is a resource's property rather than a generic transform: it knows that the API's field name and the form's field name are the same thing, and it is the only place that has to.

Error handling

Design System §23 already decided what the user sees for each class of failure. This section decides how the code gets there: one normalized error shape, one registry that turns backend error codes into explanations, and a retry policy that distinguishes safe reads from dangerous writes.

One error shape

Every failure — a dropped connection, a 422, a domain 409, a 500 — is normalized by the HTTP client (§7) into a single AppError carrying: its kind (network / validation / domain / permission / notfound / server), the HTTP status, the backend error code where one exists, field errors where they exist, and a request identifier for support. Calling code never inspects an axios error, never checks error.response?.status, and never string-matches a message. It switches on kind.

D-10 · The error-code registry is a first-class frontend asset, owned in one file

The alternative is to render whatever message the backend sent, or to handle codes ad hoc at each call site with an inline conditional. Both are rejected, and the Design System already anticipated this: it calls the registry "a design-system asset: it is how backend invariants become explainable UI instead of alert boxes."

Handling codes at call sites means COMMERCIAL_HAS_STOCK_CANNOT_REASSIGN is explained well on the agent edit form and shows a raw code on the agent list — same failure, two qualities of explanation, and the second one generates a support ticket. Rendering the backend's own message means the UI's voice, language, and recovery guidance are set by whoever wrote the exception (and Design System §27's French-first decision, O-1, would be silently violated by any English server message that leaked through).

One registry: code → message, tone, and — where one exists — a recovery path ("Le commercial détient encore du stock. Créez un retour de stock avant la réaffectation," linking to the return flow). Unregistered codes fall back to a generic message that still displays the raw code for support, so an unknown failure degrades to merely-unhelpful rather than to a blank screen. MUST: every documented backend code has an entry, and adding a backend code without adding its entry is an incomplete feature.

Retry policy — asymmetric, on purpose

Queries retry. Two attempts with exponential backoff, on network failures and 5xx only. A GET is safe to repeat by definition, and the most common failure in an office is a momentary blip that a single retry hides completely.

Queries never retry 4xx. A 403 or a 404 is a fact, not a blip; retrying it three times just delays the truth by two seconds and triples the load.

Mutations never auto-retry — MUST NOT, no exceptions. Repeated in §8 and again here because it is the rule most likely to be "helpfully" broken by someone configuring a global default. A retried deposit validation may settle twice. The frontend does not get to assume the backend is idempotent on an endpoint that moves money; it surfaces the failure and lets a human decide (Design System §23's explicit retry).

Error boundaries & logging philosophy

Boundaries exist at the route level (a crashed page must not take down the shell — the sidebar and header stay usable so the operator can navigate away) and around independently-failing panels of a WorkspacePage (Agent 360's five panels fail independently; one dead panel must not blank the other four). Boundaries catch render crashes — they are the last resort, not the error-handling strategy, and a boundary firing in normal operation is a bug to fix, not a state to design for.

Logging is discussed fully in §18, with one rule stated here because it belongs to error handling: an error that is expected (a 422, a 403, a domain 409) is not reported as an application error. Those are the system working. Reporting them buries the genuine 500s in noise, and a monitoring channel nobody trusts is a monitoring channel nobody reads.

The 401 session policy, stated explicitly

The error table above assigns 401 to the HTTP client. Because Discovery found this behavior entirely undefined in the legacy build — "a 401 mid-session has no defined UX" — the policy is spelled out here rather than left to be inferred from a table row.

RuleBehavior & rationale
No invented refresh flow The backend's authentication model does not expose a frontend refresh-token flow (Discovery §4). The frontend therefore MUST NOT invent one — no silent re-authentication, no token renewal on 401, no background refresh timer. A refresh mechanism the server does not support cannot work, and an attempt to build one produces retry storms against an endpoint that will never succeed. The absence of refresh is a backend contract, and the frontend honors it rather than papering over it.
A confirmed 401 ends the session On an authentication 401, the centralized HTTP/session layer clears the local session (the single session module, §14) and redirects to login. Centrally — a 401 handled per call site is a 401 forgotten at most call sites, and the result is a half-logged-out app still rendering a stale user in the header.
Preserve the return path, when safe The route the operator was on is preserved so that logging back in returns them to their work rather than to Overview. "When safe" is a real qualifier: only in-app paths are preserved, never a caller-supplied redirect target — an unvalidated return path is an open-redirect waiting to be used, and it is not worth the convenience.
403 is not session expiry MUST NOT be conflated. A 401 means "we do not know who you are"; a 403 means "we know exactly who you are, and you may not do this" (§6). Treating a 403 as expiry logs an operator out of a working session for clicking one thing they lacked a permission for — infuriating, and it teaches people that the app randomly signs them out. A 403 renders the calm permission state (Design System §23) and the session continues untouched.
One redirect, one clear A dashboard page fires several requests at once — a workspace fires five. When a session expires, they all fail with 401 simultaneously. The session teardown MUST therefore be idempotent and collapse concurrent 401s into a single clear-and-redirect: the first one wins, the rest are absorbed. Without this, five parallel failures produce five redirects and five session clears, which manifests as a flickering navigation loop — a bug that is very hard to diagnose and trivial to prevent by deciding it once, here.

Introducing session refresh later is a coordinated change, not a frontend one

If token refresh or rotation is wanted, it is a backend and frontend architectural change made together — the server must issue and honor refresh credentials before any client code can use them. It MUST be documented by an ADR (§19) covering both sides, and it is coupled to the token-storage decision (D-16 / ADR-0001): a move to cookie-based auth changes the refresh story and the CSRF posture (§17) at the same time. These are one decision wearing three names, and taking them separately is how a codebase ends up with half a refresh flow.

Request correlation

One reference, carried end to end

Capture: the HTTP layer (§7) captures the request/correlation identifier the backend returns when one is available, on both success and failure paths. Preserve: AppError carries it as a first-class field — it is not dug out of a header at the point of display, because by then the response object is long gone.

Surface: user-facing error states MAY expose it as a support reference — the "reference identifier for support" Design System §23 already requires on unexpected failures. It is shown as an opaque reference and nothing more. MUST NOT leak infrastructure detail to the user: no hostnames, no stack traces, no internal service names, no raw backend messages (§17). The operator gets a string they can quote; they do not get a window into the server.

Correlate: the same identifier appears in structured frontend logs and in error reports (§18), which is the entire point — it is what allows "it broke at 14:32" to be joined to a specific backend request without guessing. Correlation exists to make debugging possible, and it earns its keep only if the reference is genuinely the same on both sides.

Never correlate by attaching the payload. The identifier travels; the data does not. Authentication tokens, request bodies from financial mutations, identity documents, CIN and phone numbers MUST NOT be logged or reported (§17, §18). A correlation ID exists precisely so that sensitive context does not have to be shipped off the machine in order to debug it.

D-11 · Configuration-driven patterns — with a mandatory escape hatch

Architecture §9 proposes that most resources ship as a definition (endpoints, columns, filters, form schema, permission strings) consumed by the ListPage/DetailPage/FormDrawer patterns — "zero bespoke page code, only a definition file." This document adopts that, because the alternative is hand-building fifteen near-identical CRUD trios and the Discovery report named that the single biggest multiplier of effort in the phase.

But config-driven UI has a well-known failure mode, and pretending otherwise would be the weakest moment in this architecture: the abstraction that eats the app. The first three resources fit the config perfectly. The fourth needs one custom column renderer, so a renderCell callback is added. The fifth needs a custom row action, then a conditional filter, then a special empty state — and eighteen months later the "config" is a Turing-complete DSL that is harder to read than the fifteen simple pages would have been, and only one person understands it.

The rule that prevents this: the patterns are components with slots, not engines with options. A definition configures the common case; anything unusual is passed as a composed child — real React composition, not another config key. When a resource needs something the definition cannot express, the correct move is to compose the pattern's parts directly and write the page by hand. A hand-written page is a completely acceptable outcome and MUST NOT be treated as a failure of the abstraction — it is the escape hatch working. The failure is bending the config until it can express everything.

The tripwire: a definition field whose value is a function that takes the pattern's internals as arguments is the DSL forming. Reviewers should reject it and compose instead.

Resource metadata ownership — what the definition may contain

D-11's definition is the seam between a domain and the shared patterns, which makes it the most likely thing in this architecture to quietly grow into a dumping ground. This subsection fixes what it owns — and, more importantly, what it MUST NOT.

The resource owns a definition module

Each resource declares one dedicated configuration module (in its model/, per §3) holding the declarative metadata the reusable page patterns need — and nothing else. Its legitimate contents: display labels and the resource's naming (singular/plural, per Design System §27's vocabulary registry); table column definitions including each column's priority tier (Design System §14); filter definitions; a reference to the form schema; the set of supported actions; references to the permission strings each action requires; the status→tone presentation mapping (Design System §17); and the references that bind it to its endpoints and query keys.

The exact filename is an implementation choice and is deliberately not prescribed here. What is prescribed is that there is one such module per resource, and that it is declarative.

It references contracts — it never duplicates them

The definition is a composition of contracts that already have owners, and each of those owners is stated elsewhere in this document. It points at them; it does not restate them.

The failure this prevents is subtle and expensive: a definition that copies a permission string or a route path creates a second place where that value lives, and the two will diverge. The divergence will not fail a build — it will simply render a button the API refuses, or guard a route with a permission nobody holds, and it will be found by an operator rather than by a compiler.

The ownership boundaries — each contract has exactly one owner

Routes remain owned by the resource's routes.ts (§5) — the route table is assembled from route contributions, not from definitions. Query keys remain owned by the query-key factory (§8), which is what the invalidation map (§4, D-3) references. Permission string literals remain owned by the central permission registry (§6), which is the only place such a string appears in the codebase. Validation rules remain owned by the zod schema (§10).

The definition references all four. MUST NOT re-declare, re-type, or shadow any of them. If a permission string literal appears inside a resource definition, that is the same defect §6 already prohibits, merely relocated.

And the boundary that keeps it honest: the resource definition MUST NOT become an all-purpose "god configuration" — the single object every future need gets appended to because it is the convenient place to reach. It is declarative metadata for the page patterns, full stop. Business logic, imperative callbacks, data fetching, and conditional behavior do not belong in it: those are code, and code belongs in a component or a hook where it can be read, typed, and tested. When something cannot be expressed as declarative metadata, that is D-11's escape hatch announcing itself — compose the pattern by hand rather than growing the definition to swallow it.

Promotion to shared — the rule of three, precisely stated

A component starts in the domain that needs it, and most components stay there permanently. Promotion is a deliberate act with a test to pass — not an automatic consequence of being used twice.

UsageVerdictReasoning
One domain Stays in that domain. No discussion. A single use is not evidence of anything. Building for reuse that has not been asked for is the cheapest way to produce an abstraction fitted to a hypothesis.
Two domains Does not justify promotion on its own. Evaluate whether the concept is genuinely domain-agnostic. Duplication is an acceptable — often correct — outcome here. Two uses are indistinguishable from a coincidence. An abstraction fitted to exactly two examples is general to neither: it acquires a boolean prop for each new caller until the props are a worse API than the duplication would have been. Ask the real question — "is this one concept, or two things that currently look alike?" — and if the answer is unclear, wait.
Three or more
independent domains		 A strong signal — not an automatic requirement. Promote in a pull request that names the call sites and states the shared semantics. The third independent instance is the first honest evidence of a pattern rather than a resemblance. It is still a signal to be evaluated, not a trigger to be obeyed: three superficially similar components with three different meanings should stay three components.

The promotion test — semantics first, count second, ownership as the veto

Promotion is justified by stable shared semantics, not by reuse count. The count is evidence; the semantics are the argument. The question to answer in review is not "how many places use this?" but "do these places mean the same thing by it, and will they still mean the same thing when one of them changes?" A component used in three domains for three subtly different reasons will be pulled apart by its callers within two sprints, and the shared version will end up with a mode flag per domain — which is duplication with extra steps and a shared file everyone is now afraid to touch.

The hard veto, and it overrides every count: a component MUST NOT be promoted to shared if doing so requires the shared layer to import a domain type or to understand domain-specific behavior (§2, §4). If the "shared" component needs to know what a cheque is, it is not shared — it is a domain component with three callers, and the correct move is to pass its cheque-ness in as data. This is not a style preference: it is the one boundary in §4 whose failure turns the shared layer into a second, worse copy of the domain layer, and no amount of reuse justifies breaching it.

Both errors are real, and they are not symmetric. Premature promotion creates weak abstractions — shared code fitted to too few examples, which everyone then works around rather than fixes, because it belongs to no one. Delayed promotion creates duplication — which is ugly, visible, and trivially deletable the day the pattern becomes clear. Given a genuine doubt, prefer the duplication: it is the cheaper mistake to undo. The ownership boundary matters more than the raw reuse count, and a correct duplicate is better than an incorrect abstraction.

The exceptions are the components the Architecture already declared shared (the §7 library and the §4 patterns). Those are known-shared by specification and start in the shared layer — this rule governs new discoveries, not decided architecture.

Shared hooks strategy

Hooks follow the layers exactly — which means the question "where does this hook go?" is already answered by §2 and needs no separate taxonomy. What this section adds is the rule for the one genuinely hard case: a hook that spans domains.

Infrastructure hooks

Thin React bindings over infrastructure: useSession (the current user), usePermission (the evaluator, §6). They know nothing about Miza's domains and would survive being lifted into another product.

Shared hooks

Domain-agnostic but product-shaped: useUrlFilterState (the typed reader/ writer for the query-string state that Design System §15 mandates), useDebouncedValue (the search debounce of §15's search rules), useConfirm (imperative access to the confirmation dialog).

useUrlFilterState is the load-bearing one: it is the single place the "filter state is one serializable value" contract is implemented, and every list in the app gets it for free by using the ListPage pattern.

Domain hooks

Where nearly all data access lives: useCheques, useApproveCheque, useAgentCapacity, useRestockGate. Each wraps a query or mutation and is the only way the rest of the app reads that resource. Components never call an API module directly (§7).

The cross-domain hook rule

A hook that fetches from one domain belongs to that domain. A hook that composes data from several domains belongs to the page that composes them — not to a shared "useAgent360" in the shared layer, which would force the shared layer to import from four domains and invert the dependency rule (§4) at its single most important edge.

The one sanctioned exception is already documented: the restock gate. It is a Grattage rule that Stock's forms must respect (Architecture §5), so Grattage exports the hook from its public surface and Stock imports it. This is mechanism 2 of §4, and it is deliberately the only domain→domain hook import in the application. If a second one is proposed, it needs an ADR — not because the rule is sacred, but because the first exception is always reasonable and the tenth is how the boundary dies.

Avoiding duplication — and avoiding the cure being worse

The genuine duplication risk here is data-fetching logic: two components each calling the same endpoint with slightly different keys, caching separately, and disagreeing. The query-key factory (§8) and the "one hook per resource read" rule eliminate that class entirely — there is exactly one way to fetch a cheque list, so there is exactly one cache entry for it.

The false duplication to leave alone: two hooks that look similar but serve different domains. Merging useCheques and useDeposits into a useFinancialRecords because they are structurally alike would produce a hook with a discriminating parameter, two divergent code paths inside it, and a coupling between two domains that the backend keeps separate. Structural similarity is not shared meaning — it is the most common false positive in abstraction, and §12's rule of three exists to filter it.

Utilities & services

Formatters — the single implementation of the Design System's data rules

One module, and formatting exists nowhere else

Design System §5 and §27 specify exactly how money, dates, relative dates, phone numbers, identifiers, percentages, and large numbers render, and §29 specifies how entities are presented. This module is the sole implementation of those rules. MUST: no component formats a value inline — no toFixed(2) in a table cell, no date string built by hand, ever.

Why this is a hard rule and not a preference: Design System §5 requires that an absent amount renders as — and never as 0,00 DH — "on screens that settle real debts, that distinction is financial, not cosmetic." A single inline value.toFixed(2) in one table silently violates that, in one place, invisibly, and no one finds it until an operator acts on a zero that was really a null. Centralized formatters make that violation impossible to write by accident, and they make the French-first decision (O-1) revisable in one file rather than three hundred.

ConcernRule
Constants Permission strings (§6), status→tone maps (Design System §17), error codes (§11), the per_page ceiling. Each has exactly one home, and each is data. A magic string appearing in a component is a defect — not for style reasons, but because it cannot be renamed, found, or audited.
Configuration Environment values (API base URL, environment name) are parsed and validated at bootstrap, and the app fails loudly and immediately if one is missing or malformed. It MUST NOT boot into a broken state and surface the problem later as a mysterious network error — a missing API base URL should be a startup crash with a clear message, not a hundred failed requests. Config is read in exactly one module; import.meta.env appears nowhere else.
Storage One module owns localStorage — the session store (§17) — and nothing else in the application touches it. Discovery found useAuth() reading localStorage directly on every call and the header reading it independently of the auth hook; that is two consumers of an unowned global, and it is exactly how a logout leaves a stale user in a dropdown. One owner, one interface, and the eventual migration to a different token strategy (§17) becomes a one-file change.
Helpers Live with what they serve (§3: no utils/ folder). A helper used by one resource lives in that resource. Universal ones are named for their job.

D-12 · Feature flags are build-time constants, not a runtime service

The instinct when "feature flags" appears in a spec is to reach for LaunchDarkly or to build a flag service with a backend table. Rejected for v1. This is a single-tenant internal back office with one deployment target and no A/B testing requirement, no gradual-rollout requirement, and no need to change behavior for a subset of users at runtime. A flag service would add a network dependency, a cache, a failure mode, and a vendor — to solve a problem this product does not have.

What it does have is a real need for a simpler thing: a way to merge incomplete work safely, and a way to keep UI dark for backend endpoints that do not exist yet — Companies and Suppliers have no controllers (Discovery §6; Architecture §5 marks their screens blocked). A typed, build-time config object handles both.

The rule that keeps flags from becoming permanent debt: every flag MUST be created with a stated removal condition ("remove when the Companies controller ships"). A flag without an expiry is a permanent branch in the code, and permanent branches double the states every future change must be tested against. Flags are scaffolding, and scaffolding comes down.

Feature-flag lifecycle

A flag is a branch in the product that lives in the codebase. Two flags are four code paths; five are thirty-two, and nobody tests thirty-two. That combinatorial cost is why a flag must be born with its own death already specified.

Every temporary flag MUST haveWhy it is mandatory
A named owner A person, not a team. Flags with no owner are removed by nobody, because everybody assumes somebody else knows whether it is still needed. The owner is who gets asked at cleanup.
A creation reason What it exists to enable — "the Companies UI is built but the controller has not shipped." Six months later, the reason is the only thing that tells a stranger whether the flag is still doing anything. A flag whose reason no longer holds is already dead code with a switch on it.
A removal condition or date The concrete event that makes it obsolete ("remove when the Companies controller ships"), or a date. This is what converts a flag from an open-ended branch into a task with an end.
A documented default per environment Its value in development, staging, production, and test — stated in one place, reviewable in a diff. An undocumented per-environment default is how a half-built screen reaches production: not by decision, but because nobody wrote down what the flag was supposed to be there.
A cleanup task Filed when the flag is created, not when it is forgotten. Rollout complete means the flag and both of its branches are collapsed to the surviving one — deleting the flag while leaving the dead branch behind is not cleanup.

Expired flags are debt, and product configuration is not a flag

An expired flag MUST be removed, not left permanently in place. A flag whose removal condition has been met and which still exists is technical debt in its purest form: it costs a code path, a test matrix dimension, and a moment of confusion for every future reader, in exchange for nothing at all. "It's harmless, leave it" is how a codebase accumulates twenty of them, at which point the real behavior of the product in production is knowable only by reading a config file.

Long-lived product configuration is not a temporary feature flag, and MUST NOT be modeled as one. The distinction is intent, and it is load-bearing: a flag is scaffolding around an in-progress change and is expected to die; configuration is a permanent, legitimate difference in how the product behaves and is expected to live. Filing permanent configuration as a flag means it inherits a removal condition it will never meet, and it pollutes the flag list until the list stops being actionable — the flags that genuinely need removing become invisible among the ones that never will. Model permanent behavior as configuration (below); model temporary branches as flags.

Environment configuration strategy

Four environments, one rule: differences between them are declared in configuration, never discovered in the code.

EnvironmentPosture
Development Verbose logging (§18), relaxed defaults, error reporting off. The only environment where developer convenience is a legitimate input to a configuration decision.
Test Deterministic and hermetic: the API base URL points at MSW (§16), never at a real backend; error reporting and analytics are off; timing-sensitive behavior (debounce, retry backoff) is configurable so tests do not have to wait on real durations. A test run MUST NOT be capable of reaching a production API — that is a configuration guarantee, not a discipline.
Staging Mirrors production as closely as possible — same build, same defaults, same conservatism, differing only in endpoints and its environment marker (the staging banner of Design System §20). Staging exists to catch what production will do; every place it is configured to behave differently is a place where it has stopped being able to do that. A staging environment that is "production with debug on" tests a product that will never ship.
Production Conservative by default. Errors and warnings only in logs, no debug surfaces, no verbose diagnostics, no unfinished features enabled. Where a setting's correct production value is uncertain, the safe value wins — a flag defaults to off, a feature defaults to hidden, a limit defaults to the lower bound.

Five configuration rules

(1) Configuration is validated, never assumed. Environment values enter through one validated config module (above) and nowhere else. MUST: missing or malformed required configuration fails fast at bootstrap with a clear message — never a silent boot into a broken state that surfaces later as a hundred mysterious network failures.

(2) No component branches on an environment name. if (env === 'production') inside a domain component is forbidden. Branch on a capability the config declares ("is error reporting enabled", "is the staging banner shown"), never on the environment's identity. Environment-name checks scatter across the codebase, are invisible to search when someone adds a fifth environment, and produce components whose behavior in production has never actually been rendered by anyone before a customer sees it.

(3) No secrets in frontend configuration. Ever. Anything shipped to a browser is public — minification is not concealment, and a build-time environment variable is as readable as a comment. API keys, credentials, and signing secrets MUST NOT appear in frontend config under any circumstance. If a feature appears to require one, it requires a backend endpoint instead, and that is the design conversation to have rather than a value to hide.

(4) Environment differences are explicit and reviewable. The complete set of differences between staging and production is readable in one place, in a diff, by a person. Behavior that varies by environment through scattered conditionals is behavior nobody can predict without running it.

(5) Production defaults are conservative — restated because it is the rule that gets bent when a deadline arrives. The cost of a feature reaching production a week late is a week. The cost of an unfinished financial screen reaching production early is measured in something else entirely.

Performance strategy

This is an internal back office on office networks, not a consumer site competing for bounce rate. The performance that matters here is interaction performance — a table that filters instantly, a form that never stutters mid-entry — not shaving kilobytes off a first paint that happens once a day. Several conventional optimizations are therefore rejected below as premature, and one is rejected as actively harmful.

ConcernStrategy
Code splitting Per domain, not per route (§5, D-4), plus a shared chunk for the pattern library. Matches how operators actually navigate — they work inside one domain for long stretches — so the chunk they need is fetched once and every subsequent screen in that domain is instant.
Bundle strategy The dependency list is the bundle strategy. Two libraries are added by this document (§0) and every further one needs an ADR. A date library, an icon set, and a chart library will each be proposed; each MUST justify its weight, and "it's only 30kb" is not a justification when said four times. Bundle size is monitored in CI as a trend, not a hard gate — a gate gets raised the first time it fails, and then it means nothing.
Rendering The real risks in this app are (1) the serial-waterfall workspace (§8: independent queries MUST fire in parallel) and (2) a table re-rendering every row on every keystroke of the search box — which the §15 debounce and stable row identities both address. Both are structural, and neither is fixed by memoization.
Images The only images in this product are avatars and uploaded proof documents (CIN photos, cheque images). MUST NOT render full-resolution proof images in list cells — a table of fifty cheques must not download fifty multi-megabyte photographs. Thumbnails where the backend offers them; lazy-load below the fold; full resolution only in the viewer that exists to show it.
Caching The Query cache is the application cache, tiered by §8. HTTP caching is not relied upon for authenticated API responses (and Sanctum's bearer responses are not cacheable by intermediaries anyway). One cache, one set of rules, one place to reason about staleness.

D-13 · Virtualization is deliberately NOT implemented — and the number that proves it

Every "enterprise dashboard architecture" reaches for row virtualization, and it is the kind of decision that is never questioned because it sounds obviously correct. Here it is rejected, on arithmetic.

Design System §14 caps pagination at per_page ≤ 100, mirroring the API's own clamp, and prohibits infinite scroll outright. The maximum number of rows any table in this application can ever render is therefore 100 — a number React renders without measurable cost. Virtualization would add a library, break native browser find-in-page (which operators genuinely use to locate a cheque number), complicate sticky headers and the horizontal-scroll behavior §14 requires, break keyboard tab-order through off-screen rows (§10's accessibility floor), and complicate every future table feature — to optimize a bottleneck that cannot occur.

The trigger to revisit, stated concretely so this stays a decision and not dogma: if a screen ever legitimately needs to render more than ~200 rows at once — an export preview, a reconciliation view the backend cannot paginate — virtualize that screen, with an ADR. Do not virtualize the DataTable for all fifteen resources on the strength of one exception.

D-14 · Memoize on evidence, never by default

The convention — wrap every component in memo, every callback in useCallback, every derived value in useMemo — is rejected as a default. It is not free: each memo adds a comparison on every render and a dependency array that is now a correctness liability. A stale dependency array produces a component that renders old data — which in this product means an operator looking at a stale amount, and that is precisely the failure class this whole architecture is organized against. Memoization introduced to avoid a re-render that was never slow has traded a nonexistent performance bug for a real correctness bug.

The rule: memoize when a profiler shows a problem, or at three specific known-expensive seams — the DataTable's row and cell renderers, the LineItemsEditor's row list (where a running total recomputes on every keystroke), and any chart's data transform. Elsewhere: write the obvious code. React 19 is fast, and the app's slowest operation by three orders of magnitude is the network.

Testing strategy

Test what breaks, at the level where breaking is expensive. For this product that means: the money math and the mappers get unit tests, the pages get behavioral tests against a mocked API, and a very small number of end-to-end tests protect the irreversible flows. What it does not mean is a coverage percentage.

LevelScopeWhat it protects, and why it earns its cost
Unit cheap Pure logic only: formatters (§14), DTO mappers (§7), the permission evaluator (§6), status→tone maps, the error-code registry's completeness. These are pure functions with high consequence and near-zero test cost — the best ratio in the codebase. The formatter tests are effectively an executable copy of Design System §5/§27, and the mapper tests are where "absent is not zero" is enforced rather than merely written down. A test asserting that a null amount maps to null and renders as — is worth more than a hundred component snapshots.
Integration the main investment A page, rendered with its real hooks and real patterns, against an API mocked at the network boundary (MSW). Asserts behavior an operator would recognize: the filter narrows the list, the 403 renders the calm state, a 409 renders its registered recovery path, the confirm dialog shows the fresh amount, a failed submit keeps the form's values. This is where the value is. These tests exercise the actual integration of Query, the mapper, the error registry, and the pattern — the seams where this architecture's bugs will actually live — while remaining refactor-proof, because they touch no internals. Mocking at the network boundary rather than mocking the hooks is what makes them real: a test that mocks useCheques proves only that the mock works.
End-to-end few, precious A handful of flows against a real backend: login → approve a cheque; create a deposit → validate → confirm the grattage invoice settles; draft a stock movement → add lines → validate. Plus one negative path: the restock gate blocking an allocation. These are the flows where a bug costs money, and they are the only ones that prove the frontend and backend actually agree. They are slow and they are brittle, so there are few — E2E is a smoke alarm for the irreversible paths, not a substitute for the integration layer.

D-15 · No coverage target — and mocks derived from the real API contract

Coverage percentages are rejected as a quality gate. A coverage number is trivially gamed by testing the easy code, and it actively misdirects effort: 90% coverage achieved by snapshot-testing presentational components while the deposit-settlement flow is untested is worse than 40% coverage concentrated on the money paths, because it produces confidence that is not warranted. The review question is "what would this test have caught?" — not "what is the number?"

The real risk this section must address is contract drift. Integration tests mock the backend; a mock that has silently diverged from the real API produces a green suite and a broken app — the most expensive failure mode a test suite can have, because it converts the safety net into a source of false confidence. Mitigation: MSW fixtures MUST be derived from the backend's documented contract (the maintained Postman collection and the API endpoint docs), not hand-invented from what a developer assumed the response looked like, and the E2E layer exists precisely to catch the drift that the mocks cannot. Where a backend response shape changes, the fixture is updated in the same pull request — a fixture is part of the API contract, not part of the test's convenience.

Test organization

Unit and integration tests live beside the code they test, inside the resource folder — so a feature is deleted, moved, or reviewed as one unit (§1's deletability principle applies to tests too; a mirrored __tests__ tree is a second folder structure to keep in sync, and it never stays in sync). Shared MSW handlers, fixtures, and render helpers live in src/test/. E2E specs live outside src/ entirely, because they test the product, not the code.

Security considerations

The frontend of a financial system is not a security boundary — and the most dangerous thing this document could do is let anyone believe otherwise. This section states what the frontend assumes, what it must never assume, and the one inherited risk it explicitly accepts.

The first principle: the client is not trusted, including by us

Every permission gate (§6), every disabled button, every hidden nav item, and every zod schema (§10) is a usability feature. None of them is a security control. All of them can be bypassed by anyone with a browser console, and the architecture assumes they will be.

The backend is the only authority. It enforces every permission with permission:* middleware, and it enforces every business invariant in the service layer. If a frontend gate is ever the only thing preventing an action, that is a backend vulnerability with a frontend symptom, and it MUST be reported as such rather than patched with a better gate.

AreaAssumption & rule
Authentication Sanctum bearer tokens, obtained from /auth/login, sent as an Authorization header. There is no refresh-token flow (Discovery §4). A 401 therefore means the session is over: the client terminates it centrally, clears the store, and redirects to login preserving the return path (§11). The token is never sent anywhere but the API's own origin.
Authorization Permission strings from the login payload, evaluated client-side for rendering only (§6). Never roles (D-5). Never trusted for enforcement (above).
Sensitive data The product handles CIN numbers, phone numbers, identity documents, and monetary amounts. MUST NOT appear in: URLs (record IDs only — never a CIN as a query param, where it lands in browser history and any proxy log), logs, error-reporter payloads (§18), analytics events, or localStorage beyond the session payload the backend itself returns.
XSS The primary threat, because it is the one that turns the accepted token-storage risk (below) into a real compromise. React escapes by default; the rules that keep that true are absolute. MUST NOT use dangerouslySetInnerHTML anywhere in this application — there is no requirement in the entire product that needs it, so a request to use it is a design error, not a security exception to negotiate. Never render backend-provided HTML. Never build a URL for an href from unvalidated input. Dependency hygiene (audited installs, few dependencies — §15) is part of the XSS posture, because a compromised transitive dependency has the same reach as our own code.
CSRF Not applicable, and deliberately so. Bearer tokens in an Authorization header are not ambient credentials: a cross-site form post cannot attach them, which is the entire mechanism CSRF depends on. No CSRF token ritual is needed, and adding one would be cargo-cult. This changes the day auth moves to cookies — see D-16 — at which point CSRF protection becomes mandatory, and that dependency MUST be part of that decision.
File uploads Client-side type and size checks (Design System §12: images/PDF, 2MB documents, 5MB proofs) are UX — they fail fast and explain. The backend re-validates everything; the frontend never treats its own check as sufficient.

D-16 · Keep the token in localStorage for v1 — a consciously accepted risk, isolated for exit

The correct security answer is well known: bearer tokens in localStorage are readable by any JavaScript on the page, so a single XSS becomes a full session compromise, and the token remains valid until manually revoked because there is no refresh/rotation flow. An httpOnly, SameSite cookie would eliminate the exfiltration path entirely.

The decision is nonetheless to keep localStorage for v1, and it is important to be honest about why, because "we'll fix it later" is how this becomes permanent. Moving to cookie auth is not a frontend change — it is a backend change (Sanctum's stateful cookie mode, CORS and CSRF configuration, session handling across three tokenable models) landing in a backend that has just completed Phase 5.10 and is stable. Forcing that change now trades a well-understood, mitigated risk for a poorly-understood one, in the middle of a frontend build, on an internal tool behind an office network. The Discovery report reached the same conclusion — "standard for this app's size, but worth a conscious note."

What makes this an acceptable decision rather than negligence, and what MUST hold: (1) The XSS rules above are treated as absolute, because they are the compensating control — the entire risk is conditional on an XSS existing. (2) All token access is isolated behind the single session module (§14), so the migration is a one-file change on the frontend rather than a search-and-replace across the app. (3) This is recorded as ADR-0001 with an explicit revisit trigger: any external exposure of this dashboard beyond the internal network, or the first backend cycle with capacity for auth work, whichever comes first.

Observability

The question observability answers is "did an operator hit a problem we don't know about?" — not "how many people clicked this button." For an internal financial tool those are very different questions, and conflating them is how a product ends up shipping its clients' phone numbers to a third-party analytics vendor.

Logging

One logger abstraction; console.* MUST NOT appear in product code. Not for tidiness — because a console call cannot be levelled, filtered, scrubbed, or turned off in production, and because "temporary" debug logging of a request payload is exactly how a CIN number ends up in a browser console during a screen-share.

Development logs verbosely; production logs warnings and errors only.

Error reporting

Unexpected errors (5xx, render crashes caught by a boundary) are reported to an error service with the request identifier that §11 already carries — so a user saying "it broke" and the recorded event can actually be joined, which is the only reason to have a reference id at all.

Expected errors are not reported (§11): a 422, a 403, a domain 409 are the system working. Reporting them drowns the real 500s and trains everyone to ignore the channel.

PII scrubbing — a MUST, not a setting

Error payloads are scrubbed before they leave the browser: no tokens, no CIN, no phone numbers, no identity-document URLs, no request bodies from financial mutations. Scrubbing at the reporter's server-side is not sufficient — by then the data has already left the building, and the vendor's retention policy is now part of this product's compliance surface.

Analytics boundary

No third-party product analytics on financial screens. This is a bright line. If usage insight is ever needed (which screens are used, where operators stall), it is collected as coarse, anonymous navigation events, with no record identifiers, no amounts, and no names — and it requires an ADR that states exactly what is collected. A behavioral analytics SDK that captures DOM content will capture cheque amounts and client names by default, and it must never be installed here.

Correlation is what makes these logs worth keeping

Every structured log entry and every error report carries the backend correlation identifier that §11 captures and AppError preserves — the same reference the operator is shown as a support code. That single shared string is what turns three separate systems (the user's report, the frontend's error record, the backend's request log) into one traceable event. Without it, a production investigation begins by guessing which of four hundred requests in that minute was the one that failed.

The scrubbing rules above are not weakened by correlation — they are what makes it necessary. Because payloads, tokens, documents, and personal data MUST NOT leave the browser (§17), the identifier is the only thing that links a frontend error to its backend context, and the sensitive detail stays where it already safely lives: in the backend's own logs, behind its own access controls.

Performance monitoring — measure the thing that matters

Standard web-vitals are collected, but they are not the metric. The number that reflects this product's actual quality is time from navigation to an interactive, populated table — because that is the operator's unit of work, repeated hundreds of times a day. Track it per domain. A regression there is a real regression; a 40ms shift in a first-paint metric on an internal tool that stays open all day is noise, and chasing it is how teams spend a sprint optimizing something nobody feels.

Architectural decision records

This document will be wrong about something within six months. That is not a failure — it is the normal fate of every architecture, and the only question is whether the correction is recorded or merely applied. An ADR is how a rule changes without the reasoning being lost.

When an ADR is required

Adding a dependency. Changing or violating a rule in this document (including via a lint suppression, §4). Introducing a new cross-domain coupling (§13). Adopting a pattern this document rejected — optimistic updates (D-7), a state library (D-8), virtualization (D-13), a state machine (D-9), cookie auth (D-16). Any decision a future developer would otherwise have to reverse-engineer from the code.

What an ADR must contain

Context (what forced the decision), the options considered and rejected with reasons, the decision, and its consequences — including the ones we dislike. An ADR that lists only the chosen option is a changelog entry, not a decision record: the rejected alternatives are the entire value, because they are what stops the next person re-proposing them in a year.

How they live

Numbered, dated, append-only markdown in docs/adr/. Status: Proposed → Accepted → Superseded (by ADR-n). Never edited after acceptance and never deleted — a superseded ADR is the most valuable kind, because it records a road already travelled and why we came back.

The relationship between this document and the ADR log

This specification is the baseline: decisions D-1 through D-16 are its content and are considered accepted on approval. The ADR log records everything after it. Consequently: MUST: any change to a rule in this document requires an ADR that explicitly supersedes the numbered decision it changes, and a pull request updating this document to match. Two sources of truth that disagree are worse than either alone — and an architecture document that quietly stops describing the code is how a codebase becomes undocumented while appearing documented.

ADR-0001 is already written by §17: the localStorage token decision, with its revisit trigger. It is the model for the format.

Future scalability

The test of this architecture, exactly as in Design System §25, is the thing that does not exist yet. Here is what each kind of growth actually costs — and one thing that is refused.

GrowthWhat it costs
A new resource
hours		 One folder in its domain, following the shape in §3: an API module, a mapper, a query-key factory, a resource definition, a route contribution, permission strings. It picks one of the five existing page skeletons. No shared file changes. This is the case that must stay cheap, because it is the case that happens most.
A new domain
days		 One folder under domains/, its resources inside it, its permissions in the registry, its nav group in the sidebar (which renders itself from the registry — §6), and its statuses mapped to the six existing badge tones (Design System §17). The only shared file touched is the invalidation map (§4), and only if the new domain's mutations affect others' caches — which is a genuine business coupling that should be declared in one place.
A new role
near zero		 Nothing, structurally (§6). No component branches on a role; routes and nav carry permission strings. Granting managers dashboard access is a backend permission assignment plus, at most, new registry entries for genuinely new capabilities. This is the direct payoff of D-5.
A new API / backend change
contained		 A renamed field or a normalized status vocabulary changes one mapper (§7, D-6) — the whole reason the anti-corruption layer exists. The Companies and Suppliers controllers, when they ship, unblock screens already designed: a definition, a route, a flag removal (§14).
A second frontend
anticipated		 The backend already serves three tokenable models (User/Admin, Agent, Client) through one /auth/login. A future commercial-facing or client-facing app is a separate application that reuses this one's ideas, not its bundle — different users, different devices, different workflows. Nothing here obstructs it, and nothing here should be contorted in advance to accommodate it.

D-17 · Micro-frontends: no, and not later either

The section brief asks whether the architecture supports micro-frontends "if ever needed." The honest architectural answer is that it does — the domain boundaries, public surfaces, and dependency rules of §4 are precisely the decomposition that would make extraction possible — and that it should not happen.

Micro-frontends solve an organizational problem: multiple teams needing to deploy independently without coordinating a release. Miza has one frontend team and one deployment target. In exchange for solving a problem this project does not have, they impose runtime integration complexity, duplicated dependencies across bundles, cross-application state and auth plumbing, and — most damaging here — the near-certain fragmentation of the Design System, whose central promise (§1: "one status vocabulary, one money format, one filter pattern") depends on a single shared implementation. A federated StatusBadge in three bundles at three versions is exactly the outcome the Design System exists to prevent.

The modular monolith already delivers the benefit people actually want from micro-frontends — independent, deletable, boundaried domains — without paying the distribution tax. If Miza one day has several frontend teams shipping on separate cadences, revisit this with an ADR. Until then, this is a solved problem being re-solved expensively.

The decay this architecture is designed to resist

Every rule here has one enemy, and it is not disagreement — it is the reasonable-sounding exception made under deadline. The shared component that imports a domain type "just for the props." The status string compared inline "just this once." The optimistic update on "the one mutation where it's safe." The flag with no removal date. Each is individually defensible, and collectively they are how an architecture becomes a folder layout with a document nobody reads.

That is why §4's boundaries are lint-enforced rather than documented, why exceptions must name an ADR rather than a reason, and why the decisions in this specification are numbered: so that breaking one is a deliberate, visible, countable act — and so the person who breaks it, correctly and with good cause, can say which rule they are breaking and why.
