Miza Dashboard Design System

The single visual and UX foundation for every dashboard module defined in the approved architecture: Network, Money, Stock, Grattage, Reference, and everything added after them. Technology-agnostic by design — it constrains what the interface looks like and how it behaves, never how it is built. The Frontend Technical Architecture document (stage 4) will bind these rules to an implementation.

Version 1.0 — proposed Inputs Discovery report (approved) · Architecture proposal (approved) · backend domain audit Status Awaiting review before stage 4

How to read this document

MUST marks a rule with no exceptions — violations are design-system bugs. SHOULD marks the default; deviating requires a stated reason in review. MAY marks an explicit freedom. Reference values (hex, px) are design values, not implementation units — stage 4 decides how they are expressed in code. Callouts marked ⚖ Decision record places where a conventional choice was challenged and the verdict, so future contributors inherit the reasoning, not just the rule.

01

Design philosophy

Miza's dashboard is an operational tool: the same operators use it every working day to move real money and real stock through irreversible steps. Every principle below exists to serve that reality — none is aesthetic preference.

Calm by default, loud by exception

The interface stays visually quiet — neutral surfaces, restrained color — so that the few things that genuinely demand attention (an overdue obligation, a pending approval, a destructive action) are the only loud elements on screen.

Why: operators scan these screens dozens of times a day. If everything is emphasized, nothing is; alert fatigue on a tool that gates irreversible financial actions is a business risk, not a style problem.

Data first, chrome last

Numbers, statuses, and names are the content; containers, dividers, and decoration exist only to organize them. When a visual element doesn't help someone read, compare, or decide, it is removed.

Why: the dominant screen type in every module is a dense table of records. Decoration competes with data for the same pixels.

Progressive disclosure

A screen shows what the common case needs; complexity is available, not present. Advanced filters stay behind an expand affordance, bulk actions appear only once a selection exists, audit and metadata (created by, modified at, history) live one level deeper, and secondary actions collapse into an overflow rather than crowding the primary path. Depth is never hidden — it is deferred, and its entry point is always visible.

Why: cognitive load is the scarce resource on a screen operators read hundreds of times a day. Every control shown to the 5% case is a control the 95% case must first read and dismiss. Deferring rarely-used complexity keeps the frequent path fast without amputating the rare one — and it is what lets a dense operational tool stay calm (see above) while remaining complete.

Consistency as a contract

The same concept always looks the same: one status vocabulary, one money format, one filter pattern, one confirmation flow. A component learned in Clients is already learned in Cheques.

Why: five domains share this shell and more will follow. Per-module visual dialects multiply training cost, review cost, and bug surface. Consistency is what makes the architecture's "one shape per repeating problem" principle visible to users.

Simplicity as scope discipline

Prefer the boring, proven pattern. New visual inventions require a documented reason an existing pattern cannot serve (§25).

Why: every novel pattern is a permanent maintenance liability across every future module. The system optimizes for the tenth module, not the first.

Accessible by standard, not by audit

WCAG 2.1 AA is the floor: text contrast ≥ 4.5:1 (≥ 3:1 for large text and UI parts), visible keyboard focus on every interactive element, complete keyboard paths through every flow, meaning never carried by color alone, touch targets ≥ 44px in compact layouts, reduced-motion preferences respected.

Why: office conditions are uncontrolled — sunlight glare, low-quality monitors, keyboard-heavy data entry. Accessibility rules double as usability rules for exactly this environment, and retrofitting them costs multiples of building with them.

Scalable by roles, not by values

Every rule in this document is expressed as a role ("surface", "primary text", "danger") rather than a raw value. Modules consume roles; only this document assigns values to them.

Why: the architecture adds domains by configuration. The visual system must add them the same way — a rebrand or dark theme (§28) must be a token re-assignment, never a module-by-module repaint.

Responsiveness, honestly scoped

Desktop is the primary optimization target, because the operational workflows this product exists to serve are desktop-centric: full-day data entry, multi-column comparison, keyboard-driven approval queues (§24).

Tablet and mobile are not second-class readers. Every workflow supported at a given width MUST be fully usable at that width — not a degraded preview of the desktop one. What varies is which workflows are supported: some advanced operational flows (the onboarding wizard, multi-line lifecycle documents) remain desktop-optimized, and the interface says so plainly rather than offering a cramped imitation. Scope is chosen deliberately; quality within that scope is not negotiable.

02

Layout foundations

One shell, five page skeletons (per the approved architecture: ListPage, DetailPage, DraftLifecyclePage, WorkspacePage, plus the approval-queue variant). This section defines the spatial rules they all share.

RuleStandardRationale
Shell Fixed sidebar (≈ 250px expanded, ≈ 64px icon-collapsed) + fixed header (≈ 56px) + scrolling content region. Only the content region scrolls. Navigation and identity stay reachable at any scroll depth; matches the shell already validated in the legacy build.
Content width Content region is fluid up to a maximum of ≈ 1440px, then centers. Inside it, prose and forms cap at a comfortable reading measure (≈ 65–75 characters); tables and workspaces MAY use the full region width. Tables benefit from width; sentences and form fields do not. Capping the region prevents 27-inch monitors from producing 400px-wide table cells with mile-long gaps.
Page padding Horizontal: 24px (compact) / 32px (medium) / 48px (expanded). Vertical: 24px top to the page header, 32px between header and first content block, 64px bottom. Predictable margins make every page feel like the same product; generous bottom padding keeps the last row of a table out of the viewport edge.
Grid Layout is composed of flexible columns with a fixed gutter (24px expanded, 16px compact) — conceptually a 12-column grid, but components claim regions ("half width", "third width"), never raw column counts. Region-based thinking survives breakpoint changes; column-count thinking breaks at every one of them.
Section spacing 32px between sibling sections of a page; 48px before a section that starts a new concern (e.g. the audit trail after the data body). Spacing is the primary grouping device (see hierarchy rule below); two distinct values are enough to encode "related" vs "new topic".
Card spacing 16px gap between cards in a grid; 16–20px internal card padding; cards never touch edge-to-edge and never nest (§16). Uniform card rhythm is what makes composed pages (Overview, workspaces) read as designed rather than assembled.
Visual hierarchy Exactly four nesting levels: page → section → card/panel → field/row. Hierarchy is expressed by spacing and typography first, borders second, background shifts third — shadows are reserved for elevation (§8), never hierarchy. Four levels cover every screen in the architecture. A fifth level is a signal the page is doing too much and should split (usually into a tab or a drawer).

Page header standards

Every page opens with the same header block, so an operator entering any screen in any module already knows where they are, what they can do, and what state the record is in. The elements below are the complete vocabulary; a page uses the ones it needs and never invents a seventh.

ElementRule & visual weight
Breadcrumb Above the title, caption role, tertiary color — the quietest element in the block. Shows the path back through the hierarchy (module → list → record), not the full site tree. Present on detail and workspace pages; omitted on top-level lists, where it would only ever say the page's own name.
Title The page-title role (§5), exactly one per page, and it names the record or the job — "Aït Benhaddou, Yassine" or "Validation des dépôts", never a generic "Détail". It is the single loudest element in the header and the anchor everything else aligns to.
Status A badge (§17), sitting on the title's baseline immediately to its right. Directly adjacent because on a record page the status qualifies the title: which agent is not a separate question from whether that agent is blocked. Never relegated below the description, where it reads as metadata rather than as a fact about the thing named.
Description One line under the title, body role, secondary color. Carries the record's identifying context (key metadata, parent entity) or the page's purpose — whatever the operator would otherwise have to scroll to confirm they opened the right thing. Optional, and genuinely omitted when the title says everything; a description that restates the title is noise.
Primary action Right-aligned on the title's line, primary variant, at most one (§11). It is the page's reason for existing — create the record, validate the document, approve the queue item. Hidden entirely when permission or lifecycle state forbids it; never shown disabled without a stated reason (§10).
Secondary actions Left of the primary, secondary or ghost variants. Beyond two, they collapse into a single overflow menu rather than growing a button row — per §1's progressive-disclosure principle, a header that offers eight equal actions has communicated no priority at all. Destructive actions live only in that overflow (§11).

Header hierarchy in one sentence

Reading order is where I am (breadcrumb) → what this is (title + status) → what I need to know (description) → what I can do (actions, weighted by priority). Visual weight MUST follow that order, and the block MUST NOT scroll away into a sticky bar that keeps the actions while dropping the identity — an approve button detached from the name of what it approves is a defect, not a space saving.

03

Design tokens

Tokens are the contract between this document and any implementation. They are organized in three layers; modules may only consume the top one.

Layer 1 — Primitives

Raw values with no meaning: the full color ramps, the numeric spacing steps, the font sizes. Never referenced directly by any screen or component.

Layer 2 — Semantic roles

Meaningful assignments of primitives: "surface", "text-secondary", "border-strong", "danger", "space-section". This is the layer themes re-map (§28) and the layer this document mostly defines.

Layer 3 — Component tokens

Per-component bindings ("button-primary-background", "table-header-text") defined only where a component must deviate from the plain semantic role. Kept deliberately rare.

Token familyContents (conceptual)
ColorBrand ramp, neutral ramp, surface roles, text roles, border roles, five semantic tones (§4, §17).
TypographyOne family stack, nine named text roles with size/weight/line-height, plus a numeric variant flag (§5).
SpacingEight steps on a base-4 scale with named usage tiers (§6).
RadiusFive steps from control to overlay plus "full" (§7).
Shadow / elevationFour elevation levels, each pairing a shadow value with a stacking meaning (§8).
Z-index layersNamed stacking strata, not numbers: content < sticky < navigation < dropdown < drawer < modal < toast < tooltip. Components claim a stratum; only the token file knows the integers. MUST: no component ever declares an ad-hoc z-index.

The one rule that keeps tokens honest

If a screen needs a value that no semantic token provides, the token system is extended through review — the screen never hardcodes the value. One-off values are how design systems die; this rule is the immune response.

Semantic color is a promise

A semantic hue MUST always mean its semantic. Green never decorates a chart that isn't about success; red never highlights a marketing number. The moment a semantic color appears decoratively anywhere, operators stop trusting it everywhere — and this dashboard's entire alert model depends on that trust. Data-visualization palettes are a separate, non-semantic set (defined when the Reporting specification needs them).

05

Typography

One family, nine roles, and strict numeric rules — because this product is mostly numbers.

Decision: system font stack, not a webfont

Challenged: "professional dashboards need a branded webfont." Verdict: rejected for v1. The native UI stack (Segoe UI on the Windows fleet this back office actually runs, with platform equivalents elsewhere) renders French diacritics and Arabic-script fallbacks flawlessly, costs zero load time, and can't fail into a fallback mismatch. A brand face can be introduced later as a token re-assignment (§3) — display roles only, without reflowing data screens.

Numeric values

MUST: tabular (fixed-width) numerals everywhere digits align vertically — tables, stat tiles, line-item totals, countdowns. Proportional numerals are acceptable only inside running sentences. Quantities are integers; identifiers (CIN, cheque number, account number) are text, never formatted as numbers.

Monetary values

MUST: Moroccan French format — space as thousands separator, comma as decimal, always two decimals, currency after the amount: 12 500,00 DH. Right-aligned in tables. Negative amounts use the danger text color and a true minus sign. Absent is not zero: missing values render as — (tertiary color), never 0,00 DH — on screens that settle real debts, that distinction is financial, not cosmetic.

Data formatting — the value vocabulary

Money earns the strictest rules above, but every value type an operator reads repeatedly needs one canonical shape. A value that renders three ways across three screens forces re-reading each time, and re-reading is where transcription errors are born. These rules are conceptual — §27 owns the locale and the copy.

Value typeRule
Phone numbers Displayed grouped for reading, never as an unbroken digit run — grouping is what makes a number checkable against a paper form or a screen. One grouping convention product-wide, applied at render; the stored and copied value stays canonical, so copying a phone number yields something dialable, not something formatted. Tabular numerals in tables so columns align. Never truncated: a partial phone number is useless, and it is a primary search key (§15).
Dates One absolute format everywhere (§27), never locale-inferred from the browser. Day-precision for business dates, minute-precision timestamps only where the time genuinely matters (audit trails, deadlines). Tabular numerals; dates in a column are compared vertically, and that only works when digits align.
Relative dates Permitted only where recency itself is the information — activity feeds, "last updated" captions — and always with the absolute value one hover away (§27). MUST NOT be used for deadlines, financial dates, or any value an operator might act on or repeat aloud: "il y a 2 jours" is not a date you can put on a cheque. Relative time is a convenience for scanning, never a system of record.
Identifiers CIN, cheque numbers, receipt numbers, account numbers, reference codes: text, not numbers. No thousands separators, no rounding, no locale formatting, no arithmetic — an identifier that picks up a space separator has been silently corrupted on screen. Monospace/tabular treatment so character positions are comparable, selectable and copyable in full, middle-truncated with the complete value on hover when space forces it (§27).
Percentages One consistent precision (whole numbers unless the fraction is decision-relevant; never more than one decimal). The % stays attached to its number. Where the percentage summarizes an underlying pair that is what the operator actually acts on — capacity used, gate thresholds — the pair travels with it ("72 % · 36 000 / 50 000 DH"): a percentage alone tells you the shape of a problem but never its size.
Large numbers Thousands separators always (§27); abbreviation (12,5k) is prohibited on any monetary or quantity value an operator may act on, reconcile, or transcribe — abbreviation destroys precision to save four pixels on the screens where precision is the product. It is permitted only in KPI stat tiles where the exact value is available on hover and the number is a trend signal, not a figure. Counts follow the same separator rules as amounts, so a column of quantities and a column of money scan identically.
Absent values The em-dash rule from money generalizes to every type: absent renders as — in tertiary color, never as 0, an empty cell, "N/A", or "null". Empty and zero are different facts, and only one of them means someone still has work to do.

Spacing system

Base-4 scale, eight steps. The scale is small on purpose: with few choices, spacing decisions become automatic and pages built by different people align.

4px
Micro: icon-to-label gap, badge internal padding.
8px
Tight siblings: chips in a row, label-to-input, cell padding (vertical).
12px
Related items: fields inside a group, buttons in an action row.
16px
Default: card grid gap, card padding, form field vertical rhythm.
24px
Group separation: between form sections, filter bar to table, gutters.
32px
Page sections: between sibling sections, page header to content.
48px
Topic breaks: before a genuinely new concern on the same page.
64px
Page bottom padding; hero-scale breathing room where it exists.

Nesting rule

Space between containers MUST be greater than or equal to space inside them. A 16px-padded card separated by 8px gaps reads as one blob; proximity is the grouping signal and this rule keeps it truthful.

Interface density

Density is how much information a region presents per unit of space — the same tokens, drawn from different tiers of the scale above. It is chosen per surface by the task that surface serves, never per taste, and it is consistent across every module: a table is never cosy in one domain and cramped in another.

DensityCharacterWhere it applies — and why
Comfortable Generous padding, larger vertical rhythm, roomy control spacing. The most air of the three. Forms, drawers, modals, detail pages, onboarding. These are deliberate surfaces: one thing is being entered or decided, accuracy beats throughput, and the cost of a mis-click is a wrong amount on a real cheque. Space here is error prevention, not decoration — it separates targets and lets a label bind unambiguously to its field.
Compact The default. Moderate padding, ~44px rows, controls at their standard size. Data tables, list pages, filter bars, the application shell. The workhorse density and the one this system is tuned around (§14). It fits a useful number of records in a viewport without forcing the eye to work — the balance point between "how many rows can I see" and "can I still scan a row without losing my place".
High Tight padding, reduced row height, compact controls. The floor, not a target. Dashboard and workspace panels, summary cards, reference tables, dense read-only rollups (Agent 360, Overview). Surfaces whose job is comparison at a glance across many small facts, where scrolling to see the shape of things defeats the purpose. Permitted only where the content is predominantly read, not edited or clicked through.

Density constraints

(1) Density is a property of a surface, not of a component — one page does not mix three densities; a table is compact everywhere it appears. (2) Density MUST NOT compromise the accessibility floor (§1): touch targets, hit areas, and focus rings hold their minimums at every density, so High density reduces padding, never reachability. (3) High density is never applied to a destructive or irreversible control — crowding the buttons that reject a cheque is how a wrong row gets rejected. (4) Density is not user-configurable in v1: a density toggle multiplies the QA surface of every screen to solve a problem the defaults above should solve properly the first time.

Naming note: "compact" appears with three distinct meanings in this document and they are not interchangeable — the density here (information per unit space), the button size (§11, 30px in tables and toolbars), and the width class (§24, < 768px). Context disambiguates; a component spec MUST say which it means.

07

Border radius

Radius encodes the class of an element: the bigger the container, the softer the corner. Five steps plus full.

Control4px — checkbox, tag
Input7px — fields, buttons
Container10px — cards, tables
Overlay14px — drawers, modals
Fullpill — badges, avatars

MUST: nested elements use a smaller radius than their container (an input inside a card: 7 inside 10). Equal or larger inner radii create visibly misaligned corner curves — the most common "something feels off" defect in composed UIs.

08

Shadows & elevation

Elevation answers one question only: is this element floating above the page? Four levels.

LevelShadowUsed by
0 — FlatNone; borders do the work.Everything by default: page regions, form fields, table internals.
1 — RestingBarely-there soft shadow.Cards and table containers — enough lift to separate from canvas, no more.
2 — RaisedClear directional shadow.Transient anchored elements: dropdowns, popovers, date pickers, the sticky table header once scrolled, the bulk-action bar.
3 — OverlayStrong shadow + dimmed scrim.Drawers and modals — the only level that blocks interaction behind it.

When elevation must NOT be used

Never for emphasis ("make this card pop"), never for hierarchy inside a surface, never on hover of non-floating elements (rows highlight with surface-inset, they do not levitate), and never stacked — two Level-3 overlays simultaneously is a design error; close the first or use a full page (§18). If a level jump is used for importance anywhere, elevation stops meaning "floating" everywhere.

09

Iconography

Philosophy & style

One icon library for the entire product, outline style, uniform ~1.5px stroke. Icons are labels' assistants, not their replacements. MUST: no emoji in product UI — emoji render differently per platform and carry uncontrolled tone on financial screens (these documents use them; the product does not).

Sizes

Three functional sizes: 16px inline with text and inside table cells; 20px in buttons, nav items, and form controls (the default); 24px in page headers and modal titles. One illustrative size, 40–48px, exclusively for empty and error states. No other sizes exist.

Functional icons

An icon that triggers or indicates something MUST have an accessible name, and — if it is the only visual (icon-only button) — a tooltip. Domain concepts keep one fixed icon forever: the same glyph always means cheque, deposit, stock movement, client. This vocabulary is documented once and extended, never re-assigned.

Decorative icons

Purely decorative icons are marked hidden to assistive technology and are rare by policy — in a data-first product, most decoration is noise (§1). Empty states are the one sanctioned place for illustrative iconography.

10

Interaction states

Every interactive component implements this exact state set with the same visual grammar, so state recognition transfers between components.

StateVisual grammarUX reasoning
DefaultResting appearance per component spec. Must already look interactive (affordance) — hover may confirm, it never reveals.
HoverOne-step background shift (surface → inset, or color → hover step). No motion, no elevation change. Confirms the pointer target before commitment. Touch devices never see it, so nothing may depend on it.
Focus2px primary-color ring, 2px offset — identical on every focusable element, always visible, never removed. Keyboard operators are the power users of this product (data entry, approvals). One universal ring means the eye finds "where am I" instantly on any screen.
ActivePressed step of the color ramp; ≤ 1px translation at most. Sub-100ms confirmation that the press registered — the cheapest trust signal there is.
SelectedPrimary-subtle background + primary border or edge indicator. Persistent, unlike active. Selection is state, pressing is an event; they must not share a look — rows, nav items, and chips all follow this.
DisabledReduced contrast, muted colors, interaction blocked. Where the reason isn't obvious, a tooltip explains it. "This action exists but is unavailable now." The explanation rule matters here: a disabled Validate button with no stated reason generates support requests, not compliance.
ReadonlySurface-inset background, full-contrast text, no input affordance. Selectable and copyable. Distinct from disabled by design: readonly says "this value matters, you just can't edit it here" — a validated bon's line items are readonly, not disabled. Users must be able to read and copy them at full contrast.
LoadingComponent keeps its exact footprint; content swaps to spinner/skeleton; interaction suspended. Layout shift during load is disorienting and causes misclicks on the thing that moved. Loading is a state of the same element, not a different element.

Keyboard navigation

The focus state above says where you are. These principles govern how you move — and they are not an accessibility afterthought here: the people who use this product all day are data-entry and approval operators, and for them the keyboard is the fast path, not the fallback. A flow that can only be completed with a mouse is an unfinished flow.

Logical tab order

Focus moves in the order the content is read — the visual order of the page, top to bottom, primary path before secondary. Tab order MUST follow the page's structure rather than being patched with explicit overrides; when the two disagree, the layout is wrong, not the tab order. Every interactive element is reachable, and nothing that isn't interactive is.

Skipping is as much a defect as detouring: a Tab that jumps over the field an operator was about to fill teaches them to reach for the mouse permanently.

Keyboard-first workflows

The high-frequency operational paths — filling a form, moving through a filter bar, working an approval queue, entering document line items — MUST be completable end to end without a pointer, including submission and confirmation. Confirmation dialogs are part of the path: a modal that can only be dismissed by clicking breaks the flow it was meant to protect (§19).

Repetitive entry is where this pays: an operator processing forty cheques should never leave the home row to do it.

Predictable focus movement

Focus never moves without cause, and always moves where the operator is looking. Opening an overlay moves focus into it and traps it there; closing returns focus to the element that opened it (§18, §19). A validation failure moves focus to the first invalid field (§13). Async results never steal focus mid-typing — content arriving is not a reason to move someone's cursor.

The rule underneath all of these: after any action, an operator MUST be able to predict where the caret went without looking for it.

The non-negotiables

Focus is never removed or made invisible (§10) — "it looked cleaner without the ring" is the single most common way a product becomes unusable for its heaviest users. Escape closes the topmost dismissible layer and nothing else. Standard keys keep their standard meanings everywhere: Enter submits, Space toggles, arrows move within a composite control (a select, a radio group, a menu), Tab moves between controls — never within one. Custom shortcuts MAY be added later, but they are an accelerator layer on top of a system that already works with the standard keys alone; they are never the only way to reach an action.

Buttons

Button variant = action priority. One glance at a screen's buttons should reveal its action hierarchy.

Valider le dépôt Exporter Voir l'historique Rejeter Annuler le chèque
VariantPurpose & priorityUsage rules
Primary The single main action of the current context — submit, save, validate, approve. MUST: at most one per view (page, drawer, or modal). If two actions feel primary, the screen has two jobs — split it. Positive irreversible actions (validate a bon, approve a cheque) are primary plus a confirmation modal (§19) — irreversible ≠ destructive.
Secondary Legitimate alternatives beside the primary: cancel, export, save draft. Outlined on surface. The default variant when in doubt. Sits left of the primary in action rows (primary is always outermost-right).
Ghost Low-priority and repeated actions: table row actions, "view all" links, toolbar utilities. No border, no fill until hover. The only variant allowed to repeat many times on one screen — which is exactly why it must stay visually quiet.
Danger Destructive or negative-outcome actions: reject, annuler, block, delete. Filled danger for the confirming step (inside the confirmation modal); outlined danger for the initiating step (on the page). MUST never be the only visual cue — the label carries the verb ("Rejeter le chèque", not "Confirmer").
Icon button Space-constrained repeated actions: row actions, drawer close, header utilities. MUST have an accessible name and a tooltip. Minimum 32×32px hit area inline in tables, 40×40px elsewhere, 44×44px on compact/touch layouts.

Decision: no Success button variant

The requested set included a Success (green) button. Rejected. Approval actions (validate, approve, settle) are the primary action of their context — that is what the primary variant is for. Shipping a green action button creates two competing "positive" affordances per screen, and it spends the success hue on affordance when §4 reserves semantic color for state. Green tells you what happened; teal tells you what you can do. Keeping those channels separate is worth more than a prettier approve button.

Sizes & placement

Two sizes: default (36px height) and compact (30px, tables and toolbars only). Action rows: primary at the right edge, secondary to its left, tertiary/ghost leftmost or separated. Modals and drawers follow the same order in their footer.

Submission behavior

On submit, the triggering button enters its loading state (spinner replaces label, width preserved) and the whole form suspends — double-submission of a deposit validation must be structurally impossible, not merely unlikely.

Form controls

Shared anatomy for all controls: label above, control, then one reserved line for helper or error text (§13). Uniform 36px control height, input radius, border-strong outline, focus ring per §10.

Text input

Single-line, placeholder is an example — never the label. Identifier fields (CIN, cheque number, account number) render in the numeric/mono treatment and never autocapitalize or autocorrect.

Number input

Integers only (quantities). Right-aligned, tabular numerals, browser spinner arrows suppressed — they cause off-by-one errors on precise entry. Min/max mirror the backend rule and are stated in helper text, not discovered on error.

Money input

A distinct control, not a reused number input: fixed "DH" suffix inside the field, two-decimal comma format enforced on blur, right-aligned tabular numerals, paste-tolerant (accepts "12 500,00" and "12500.00", normalizes to §5 format). Amounts are the highest-stakes values in the product; their control earns its own spec.

Search

Leading search icon, one-tap clear affordance when non-empty, debounced as it types. Placeholder states the scope: "Rechercher par téléphone…" — matching the backend's actual search fields per resource, so it never overpromises.

Select

For closed lists the user recognizes (status, method, operator, ville). Single-select, typed lookahead. Rule: under 5 mutually exclusive options in a form, prefer radio (§ below) — a select hides options a glance could show.

Autocomplete (entity picker)

For records searched, not scrolled: agents, clients, products. Async search against the API, results show the entity chip (name + key metadata), selected value renders as that chip with a clear affordance. MUST be used wherever a foreign key is chosen — raw ID entry is prohibited in forms.

Date picker

Calendar popover + direct keyboard entry (DD/MM/YYYY) — data-entry users type dates faster than they click them. Range variant for filters with presets (§15). Weeks start Monday; locale per §27.

Textarea

Multi-line notes and reasons (rejection reason, cancellation note). Auto-grows to a max (~6 lines) then scrolls; character limit surfaces at 80% consumed, not at failure.

File upload

Drop zone + browse button; accepted types and size cap stated up front (mirroring backend rules: images/PDF, 2MB documents, 5MB proofs). Image files preview as thumbnails; every file row gets replace and remove actions; upload progress is determinate (§21). Multi-file flows (agent onboarding) list each requirement as its own named slot — "Photo CIN recto", "Photo CIN verso" — never one anonymous bucket.

Checkbox

Independent yes/no facts and multi-select lists. In forms, checkboxes are the boolean control (see toggle decision below). Indeterminate state exists solely for select-all headers (§14).

Radio group

2–4 mutually exclusive, always-visible choices: agent role (manager/commercial), deposit method (bank/cash), allocation type (rapped/grattage). Options carry sublabels when the choice has consequences.

Toggle switch

Reserved for immediate-effect state changes outside forms — a table view option, a live setting. ⚖ Decision: toggles inside submitted forms are prohibited: a switch communicates "this took effect now", which is a lie inside a form that hasn't been saved. Forms use checkboxes; switches act instantly or not at all.

Data tables

The workhorse. Most screens in every module are this component; these rules are the ones implementation will live inside daily.

ConcernStandard
LayoutOne table per Level-1 container: header row (surface-inset), body rows separated by border-soft lines, no zebra striping (row lines + hover are sufficient; stripes add noise at 13px density). Row height ~44px default. Full region width; when columns exceed it, the table scrolls horizontally inside its container — columns are never crushed to fit and the page body never scrolls sideways.
ColumnsText left-aligned; numbers and money right-aligned with tabular numerals; dates in one format (§27); status as badge (§17); entity references as entity chips (name + click-through), never raw IDs. First column is the row's identity (name or document number) and is the row's primary link. Column count earns its place: 7 ± 2 visible columns; more belongs in the detail view.
Column priorityEvery column declares a priority when the table is declared — it is part of the column set, not a responsive afterthought. Three tiers, and the order they survive in as width shrinks: P1 — identity and decision. Never dropped, at any width: the identity column (name or document number), the status badge, and the primary amount or quantity. These three answer "which record, what state, how much" — the questions that make a row actionable. A row stripped to P1 is still a usable row; strip further and the operator must open every record to know anything, which is slower than scrolling. P2 — operational context. Dropped only under real pressure: the dates, the counterparty or assigned agent, the method, the secondary amounts. These distinguish rows from one another and are what filtering and scanning actually use. P3 — metadata and audit. First to go: created-by, modified-at, internal references, derived counts. Per §1's progressive-disclosure principle, this tier is arguably deferred even on wide screens — its natural home is the detail view. MUST: a column's priority is fixed per resource and identical for every operator; a table whose columns disappear in a different order on two machines is a table nobody trusts.
SortingSingle-column, server-executed. Sortable headers show an affordance on hover and a direction indicator when active; the default sort is stated per resource (usually newest first) and visibly indicated — invisible default sort is how "the list is broken" tickets are born.
FilteringLives in the FilterBar above the table (§15), not in column headers. ⚖ Decision: per-column header filters rejected for v1 — they duplicate the FilterBar, fight with sorting affordances in the same 32px header, and the backend exposes curated filter params per resource, not arbitrary column predicates. Revisit only if a real usage pattern demands it.
PaginationServer-side always. Page size selector 20 / 50 / 100 — ceiling matches the API clamp (per_page ≤ 100). Shows total count and current range ("21–40 sur 187"). Page state is part of the shareable filter state (§15). Infinite scroll is prohibited: operators need stable positions ("the cheque was on page 3") and reachable footers.
Row selectionLeading checkbox column, appears only on tables that offer bulk actions. Header checkbox selects the current page (indeterminate for partial); selecting all pages is a deliberate second step with an explicit count — silent all-page selection on an action like bulk-assign is how accidents at scale happen.
Bulk actionsA contextual action bar (Level-2 elevation) replaces the FilterBar while a selection exists: count, available actions, clear-selection. Caps mirror the backend (bulk-assign: 100) and are surfaced in the count ("100 max"), not on failure.
Row actionsTrailing column: at most two ghost icon buttons for the frequent actions + an overflow menu for the rest. Destructive row actions live only in the overflow, separated, in danger color — never as a bare one-click icon.
Sticky behaviorHeader row sticks while the table scrolls (gaining Level-2 shadow when detached). On wide scrolling tables the identity column MAY stick left and the actions column right.
Empty / loadingEmpty per §22 (first-use vs filtered-empty are different states). Loading: skeleton rows preserving exact column geometry on first load; subsequent filter/page changes keep the current rows dimmed under a subtle indicator rather than flashing to skeleton — layout stability beats spinner honesty.
ResponsiveColumns carry a priority; medium widths drop low-priority columns before scrolling. Compact widths: the table scrolls horizontally with the identity column stuck, or — where a row is mostly status + amount (approval queues) — the ListPage MAY define a stacked row presentation. Tables never reflow into free-form card soup by default; a financial table's comparability is its columns.
15

Filters

Model: one FilterBar, declared per resource

Every list screen owns a FilterBar above its table: search field first, then typed filter controls declared per resource (status, agent, ville, date range, method…), mirroring exactly the filter params the backend accepts — the UI never offers a filter the API can't honor.

Global vs local

Local filters (the FilterBar) scope one list and reset when leaving the module. Global context is navigation, not filtering: "this agent's cheques" is the Agent 360 workspace scoping its panels, never a hidden global filter silently narrowing every screen — invisible scope is a correctness hazard on financial data.

Search

Scoped per resource to what the backend actually searches (client phone, receipt-number prefix…), stated in the placeholder. Debounced; contributes to the same filter state as everything else.

Date ranges

Range picker with presets: Aujourd'hui, 7 derniers jours, 30 derniers jours, Ce mois, personnalisé. Presets cover the operational questions ("what came in this week?") that are 90% of real usage; custom covers audits.

Status filters

The primary status dimension of a resource renders as segmented tabs above the table (Tous · En attente · Validé · Rejeté) — it's the filter used constantly and deserves one-click access. Secondary statuses stay in the FilterBar as selects.

Multi-select filters

Checkbox-list dropdown with count summary on the trigger ("Ville · 3"). Applied filters render as removable chips in a row under the bar, with one "Tout effacer" — active filters must be visible without opening anything, or filtered views get mistaken for small datasets.

Search behavior

Search is the most-used control in the product — an operator with a client on the phone reaches for it before anything else. Its behavior is specified here rather than left to each list, because inconsistent search is worse than no search: it teaches operators to distrust a negative result, and a distrusted "no results" gets re-run as a manual scroll through 200 rows.

RuleBehavior
Scope is stated, never guessed The placeholder names exactly what is searched, per resource, mirroring the backend's real search fields ("Rechercher par nom ou téléphone…"). MUST NOT be a generic "Rechercher…" — an unscoped box implies it searches everything, so a miss reads as "the record doesn't exist" rather than "you searched the wrong field". Search never silently searches less than it appears to. Where a resource's search covers several fields, the placeholder names them or says so.
Search feedback The control always shows which of three states it is in: idle, searching (an in-field indicator, never a page-level spinner — the page is not reloading, one region is), or settled on a result set. Once settled, the result count is visible ("12 résultats") and the active query stays in the field. A result set that doesn't say how many it found forces the operator to count rows to know whether to refine.
Debounce philosophy Search fires when the operator has plausibly stopped typing, not on every keystroke — but the pause is short enough to feel live and MUST NOT make a fast typist wait after they have finished. A submit key searches immediately, bypassing the wait, because a user who pressed Enter has told you they are done. The debounce exists to spare the backend, never to pace the operator; if it is ever perceptible as lag, it is mistuned.
No results Renders the filtered-empty state (§22), never a blank region and never the first-use state — conflating "nothing matches" with "nothing exists" is the single most damaging empty-state error in this product. It echoes the query back ("Aucun résultat pour « bennani »"), keeps the query in the field so it can be corrected rather than retyped, and offers one-click clearing. Where search is combined with active filters, it says so and lets the filters be cleared independently — an operator who cannot see that a status filter is suppressing their match will conclude the record was deleted.
Search is a filter The query is one component of the same filter state as everything else (§15): it appears in the shareable URL, it survives pagination, it clears with "Tout effacer", and it composes with filters by intersection — search narrows the filtered set, it does not escape it. There is no second, hidden search path.

Deferred: recent searches and saved queries

Recent-search history (and its sibling, saved queries) is a future consideration only — explicitly not in v1. It is recorded here because it is worth not foreclosing: since the query is already part of the serializable filter state below, recent searches would later be a list of stored values and a dropdown on an existing control, with no change to any component in this system. No design work is owed now; no design decision made now should make it expensive.

Future-proofing: filter state is a serializable value

The design constraint that makes future "saved filters" cheap without designing them now: the complete filter state (search, filters, sort, page size) is one serializable, URL-encodable value — every filtered view is shareable and bookmarkable from day one. A saved filter is then just a named copy of that value; no component in this system would change. This constraint is binding now; the feature is not.

Status badges — the unified semantic system

One badge component, six tones, and a canonical mapping that every current and future module MUST use. The audit found the backend mixes English status strings (draft, settled) with French ones (en_attente, accepter) — the badge system therefore maps by meaning, never by string, and owns the display label (§27).

Brouillon En attente En cours Validé En retard Annulé
ToneMeaningBackend statuses mapped (by domain)
Neutral Not yet real — editable, no side effects occurred. Stock docs: draft
Warning Awaiting a decision or deadline — someone's queue. Cheques: en_attente · Deposits: pending · Grattage invoices: pending · Clients: pending (OTP registration)
Info In transition, no action needed from the viewer. Reserved — no current backend status maps here; exists so future in-flight states (e.g. a processing reconciliation) don't force a new tone.
Success Confirmed good terminal or healthy-active state. Stock docs: validated · Cheques: accepter · Deposits: validated · Grattage: settled · Agents/Clients: active · Restock gate: clear
Danger Failed, refused, late, or barred — needs intervention or explains a refusal. Cheques: rejetee · Deposits: rejected · Grattage: overdue · Agents/Clients: blocked · Restock gate: gated
Muted (outlined) Terminated on purpose — historical, closed, no attention needed. Stock docs: cancelled · Cheques: annuler · Grattage: cancelled · Agents: inactive

Extension rules

(1) A status MUST map to one of these six tones before it may appear in any UI; inventing a seventh tone requires amending this document. (2) The same business meaning always gets the same tone in every module — rejected is danger whether it's a cheque or a deposit. (3) Badges always carry a text label; the dot-and-color alone is never the only signal (§1 accessibility). (4) Muted is visually distinct from Neutral (outline vs fill) because "not yet" and "no longer" are opposite ends of a lifecycle and operators filter on exactly that difference.

Drawers

The decision rule across the three containers: modal = a decision · drawer = a task · page = a workspace.

Use a drawer when…

The task is quick and the list context matters: create a client while looking at the client list, edit a ville, reject-with-reason, preview a record before committing to its page. Right-anchored, ~480px (forms) / ~640px (previews), Level-3 elevation.

Use a full page when…

The task is the destination: lifecycle documents (bon/allocation/transfer/return — draft, lines, validate), the agent onboarding wizard, any 360 workspace, anything with tabs, and any record someone will need to link to directly. If it deserves a shareable address, it's a page.

Never a modal instead, because…

Modals block the very context a task needs — you cannot check the list you're adding to, scroll it, or copy from it. A form in a modal also invites the worst failure mode: dismissing a half-filled form with one stray click on the scrim. Modals carry decisions (§19); the moment one grows form fields beyond a single reason-textarea, it should have been a drawer.

Drawer conduct

One drawer at a time — a drawer never opens a second drawer (that's a page telling you it's a page). Dirty forms require dismissal confirmation. Escape and scrim-click close a clean drawer; focus is trapped inside while open and returns to the trigger on close. On compact widths, drawers become full-screen sheets (§24).

Modals

Modals interrupt — the most expensive move in the interface. They are reserved for decisions that must not be made accidentally. Anatomy: title (the question), body (the consequence), footer (secondary cancel + the committing action). Focus trapped; the cancel action holds initial focus on destructive dialogs.

TypeDefinitionRules
Confirmation Positive but consequential — often irreversible: validate a bon, approve a cheque, settle via deposit validation. Primary confirm button. Body states the specific consequence in the domain's words: "Valider débite le stock du manager et crédite le commercial. Cette action est irréversible." Named entities and amounts appear in the body — confirming which cheque for how much is the entire point.
Warning Proceeding conflicts with an advisory state: allocating to a commercial whose restock gate is active, reassigning near a constraint. Warning icon + warning-subtle header treatment. Explains the conflict and both paths; proceed is available but never the visually dominant default.
Destructive Negative irreversible outcomes: reject, annuler, block, delete. Filled danger confirm labeled with the specific verb ("Annuler le chèque", never "OK"/"Oui"). Typed-confirmation (re-entering an identifier) is reserved for the highest tier only — actions that reverse money or destroy non-drafts. ⚖ Decision: typed confirmation everywhere was rejected: friction on routine draft deletions trains people to type ritually, which destroys the protection exactly where it matters.
Success Post-action acknowledgment. ⚖ Decision: effectively abolished. Success is a toast (§20), not an interruption — a modal demanding a click to acknowledge that things went well is hostile at operational frequency. The single exception: success that forks the workflow ("Bon validé — créer l'allocation maintenant?") MAY use a dialog, because then it's really a decision.

Notifications

Rule of thumb: toast confirms · inline explains · banner warns globally. Choosing the wrong channel is the most common notification defect — each exists for a different persistence and a different scope.

Toasts

Transient outcome confirmations of async actions: saved, validated, assigned. Top-right stack, auto-dismiss 5s, success/info tones only by default. MUST NOT carry errors the user has to act on — an error that vanishes on its own is an error the user missed. Failed background actions MAY toast persistently (manual dismiss + retry affordance).

Inline alerts

Contextual, persistent, placed exactly where the problem lives: form-level error summaries, the warning card on an agent with outstanding obligations, a gate advisory inside the allocation form. Semantic-subtle background, icon, plain-language body, optional action. They stay until the condition resolves — they are state, not events.

Page banners

Application-scope conditions spanning all modules: connectivity lost, read-only maintenance mode, staging-environment marker. Full-width under the header, one at a time (highest severity wins). Rare by definition — a product with permanent banners has moved its problems into the chrome instead of fixing them.

Loading states

IndicatorUse whenRules
Skeleton First load of structural content whose shape is known: tables, cards, detail headers. Mimics the real geometry (columns, row heights) so the page doesn't jump on arrival. Used once per view load — refreshes and refetches keep existing content dimmed instead (§14); flashing back to skeleton makes the interface feel broken, not busy.

Empty states

An empty region answers three questions: why is this empty, is that normal, what can I do. Anatomy: illustrative icon (§9) · one-line title · one-line explanation · at most one action. Five distinct cases — collapsing them into one generic "No data" is the failure mode this section exists to prevent.

First use — nothing exists yet

Names the domain and offers the creating action: "Aucun bon fournisseur. Créez le premier bon pour recevoir du stock." The action mirrors the page's primary action and appears only with the permission to use it.

Filtered empty — data exists, nothing matches

MUST be visually distinct from first-use, or filters get mistaken for missing data: "Aucun résultat pour ces filtres" + a one-click "Effacer les filtres". Never shows a create action — creating from a filtered-empty view is almost always a duplicate about to happen.

Permission empty

The region exists but the viewer can't see its content: states that access is restricted and who grants it, without dangling an action that will 403 (§23).

Healthy empty — empty is the goal

Queues (pending cheques, overdue invoices) where zero is success. Tone flips: a quiet positive confirmation ("Aucun chèque en attente") — not the sad-face treatment. An approvals queue that celebrates being empty is telling the operational truth.

Scoped empty — no data for this record

The generic "no data available" case, and distinct from first-use: the domain has plenty of data, this parent just has none of it — an agent with no cheques yet, a client with no invoices, a workspace panel on a record that hasn't transacted. States the scope plainly ("Aucun chèque pour cet agent"), never implies the domain is empty, and offers the creating action only when creating against this parent is a real, permitted operation from here.

Why it earns its own case: showing first-use copy ("Créez le premier chèque") on an agent's panel tells an operator the product has no cheques at all, which is false and alarming on a financial screen.

Which empty state appears, and when

These are evaluated in order — the first condition that holds wins. The order matters: a permission-restricted region that also happens to have no rows MUST say "restricted", not "empty", or the operator will report a data bug that isn't one.

OrderConditionState shown
1Viewer lacks permission to see this content Permission empty. Beats every other case: emptiness caused by access is not a fact about the data, and reporting it as one is a lie that generates support tickets. Never dangles an action that would 403 (§23).
2A search query or filter is active Filtered empty / no results. Beats first-use and scoped-empty, because the operator narrowed the view themselves and the fix is one click (clear). §15 governs the copy: echo the query, keep it editable, offer clearing.
3Region is scoped to a parent record that has none of this data Scoped empty. The domain isn't empty; this record's slice of it is.
4Zero is the desired outcome (queues, overdue lists) Healthy empty. Positive tone — the work is done, not missing.
5Nothing exists in this domain yet, unfiltered First use. The only case that leads with a create action, and only for a viewer permitted to create.

Empty is not the same as broken

None of these five is an error state. A region that failed to load shows the network-error state with a retry (§23), never an empty state — rendering "Aucun résultat" for a request that actually failed silently converts an outage into a business fact, and an operator who believes a client has no outstanding cheques because the endpoint timed out will act on that belief.

ClassUX rules

Responsive behavior

Three width classes, defined by what changes — not device names. Desktop is the primary optimization target because the operational workflows are desktop-centric; every other width is designed, not left to degrade by accident. Supported workflows are fully usable at every width — the width classes change how they are presented, not whether they work (§1).

ClassRangeBehavior
Expanded≥ 1200px The reference design. Sidebar expanded, multi-column grids, full tables, drawers as side panels.
Medium768–1199px Sidebar collapses to icon rail (labels on hover/expand). Card grids reduce to two then one column; tables drop low-priority columns per §14; page padding steps down. Everything remains fully operable — managers reviewing on tablets is a real case.
Compact< 768px Sidebar becomes an overlay sheet behind a menu button; drawers and modals go full-screen; touch targets enforce 44px; sticky action footers on forms; tables fall back to P1 columns with horizontal scroll for the rest (§14). Scope is stated honestly rather than implied: the workflows supported here — consulting records, searching, approving, quick edits — are fully supported, held to the same accessibility and quality floor as anywhere else, not offered as a cramped preview of a desktop screen. Advanced operational flows (the onboarding wizard, multi-line lifecycle documents) remain desktop-optimized and stay reachable, and where one is genuinely impractical at this width the interface says so and offers the path back — it never silently presents a worse version of it.

Principles over breakpoints

Content reflows at the width where it breaks, not only at the three class boundaries — the classes coordinate the big structural shifts (sidebar, drawers, columns) so the whole product transforms coherently instead of component-by-component. Nothing in this section prescribes implementation; it prescribes what a user at each width must experience.

Future scalability

The test of this document is the sixth domain — the one that doesn't exist yet. Here is why it won't require a redesign.

Everything is a role

Colors, type, spacing, elevation are semantic roles (§3). A new module consumes roles and automatically matches; a rebrand or theme re-maps roles and every module follows (§28).

New statuses map, they don't invent

§17's six tones absorbed four domains' worth of statuses including a reserved tone for futures. A new domain's lifecycle maps its states to existing tones — the one governed extension point is amending the mapping table.

New screens are pattern instances

The architecture's five page skeletons + this document's component standards mean a new resource ships as configuration: FilterBar declaration, column set, badge mappings, form spec. The design questions are already answered; only the domain content is new.

Additions are governed, not improvised

A new component or token must state which existing one was tried and why it failed — in this document, via amendment. The version log below makes drift visible; "we added it in a sprint and forgot" is the decay mode this prevents. This document is versioned with the product and changes through the same review gate as every other stage.

Motion & animation

Why this section was added

The requested structure omitted motion, but motion timing is as much a part of a coherent visual language as radius or shadow — inconsistent durations across drawers, dropdowns, and hovers is instantly perceptible as "cheap". Left unspecified, every implementer picks their own; specified once here, it costs nothing.

Durations

Two, only: fast (~120ms) for micro-feedback — hover shifts, checkbox marks, chip toggles; standard (~220ms) for structural movement — drawers, modals, dropdowns, accordions, sidebar collapse. Nothing animates longer; this is a tool, not a film.

Easing & direction

Decelerating ease for entrances, accelerating for exits; overlays move from their anchor edge (drawer from right, toast from top). Motion exists to explain where things came from — never to decorate. No bounces, no springs, no attention-seeking loops on a screen where a pulsing element implies an alert.

Reduced motion

When the user's system requests reduced motion, structural animations collapse to instant transitions with a fade at most. Binding, not best-effort (§1 accessibility).

Content & language standards

Why this section was added

Words are the design material every component spec above depends on — labels, statuses, errors, empty states all render copy. The audit found the product currently has no language decision at all: the legacy UI is English, backend messages and cheque statuses are French, domain vocabulary is French. Leaving that unresolved would ship a bilingual accident.

Decision: French-first interface (fr-MA) — needs product confirmation

Recommended and assumed throughout this document's examples. The operators are Moroccan back-office staff; the business vocabulary is already French (bon, dépôt, annuler, ville, secteur); the backend already emits French messages and even French status values. An English UI would translate the business away from its own users. All copy is centralized from day one (no hardcoded strings), so a second language later is an addition, not a rewrite — but v1 ships one language, well. Flagged as open decision O-1 for sign-off.

Theming policy

Why this section was added

Whether dark mode exists is a decision that silently shapes every color rule in §4. Deciding it explicitly — even deciding "not now" — is cheaper than discovering mid-build that half the components assumed one answer.

Decision: light theme is canonical for v1; the token structure is theme-ready; dark mode is deferred

An internal operations tool used in daylight offices gets marginal value from dark mode, and shipping it doubles the QA surface of every semantic color decision (§4's contrast guarantees must hold twice). But because every color in this system is a semantic role (§3), dark mode later means authoring one alternate role-mapping and re-validating contrast — no component changes. The rule that makes this true is binding now: no screen ever references a raw color, only roles. Deferred is not precluded; unprepared is.

Entity presentation

Why this section was added

Entity references were specified in three places independently — the entity chip (§12, §14), the entity card (§16), and the page header (§2) — without one rule saying they are the same identity, shown at three depths. Left that way, an agent would predictably acquire a different name format in tables than in autocomplete results, and "Bennani M." in one module would be "M. Bennani (Casa)" in another. Entities are the nouns of this product; a noun that changes shape per screen costs recognition on every screen.

One identity, three depths

Every entity has exactly one identity model, and every presentation is a subset of it — never a variation on it. The depth changes; the content, order, and formatting never do. This is §1's progressive-disclosure principle applied to identity.

Chip — the inline reference

How an entity appears inside something else: a table cell, an autocomplete result, a selected form value, a line item. Carries the primary identifier, the secondary discriminator, and the status only when the status affects the reading context. Always a link to the record.

Card — the summary

The entity as a first-class object in a grid or panel (§16): mark or initials, primary identifier, key metadata, one status badge. Same fields as the chip, more of them, same order.

Header — the subject

The entity as the page's subject (§2): the identifier is the page title, the status sits on its baseline, the discriminators become the description line. Nothing new appears here that the chip contradicted.

Identity fields per entity

Each entity fixes its primary identifier, its secondary discriminator, and its status dimension once, product-wide. The discriminator is chosen to answer the question an operator actually asks when two records look alike — which is why it differs per entity, and why it is never decorative.

AgentFull name Role (manager / commercial) + territory. Role is load-bearing: what an agent may do depends on it, so it travels with the name everywhere. Active · inactive · blocked (§17)
ClientFull name Phone number — the business's real key, the search key (§15), and the field that disambiguates two clients with the same name. Active · pending · blocked (§17)

Consistency rules — binding across every module

(1) One shape per entity, forever. The same entity renders with the same fields, in the same order, with the same formatting, in every module. Recognition is the whole return on this rule.

(2) Never a raw ID. An entity reference always renders as its identity, never as a database key — extending the rule already binding on forms (§12) and tables (§14) to every surface. Operators do not think in foreign keys and MUST never be asked to.

(3) Always navigable. An entity reference is always a link to that entity's record, from wherever it appears. A named entity that isn't clickable is a dead end an operator will try to click anyway.

(4) People and organizations get a mark; things do not. Agents, clients, companies, and suppliers may carry initials or an avatar. Products and documents carry their fixed domain icon (§9) instead — initials on a product are noise dressed as identity.

(5) Status qualifies the name, and never replaces it. A blocked client still renders their name at full contrast with a badge beside it. Muting the identity of a blocked or inactive entity is how a record becomes unreadable exactly when someone is investigating why it was blocked.

(6) Historical references survive their subject. An entity that has been deactivated, blocked, or removed still renders in full in the records that reference it — a settled cheque names the agent who deposited it forever. Such references show the entity's terminal state; they never render blank, "—", or an ID. Financial history that loses its nouns is not an audit trail.
