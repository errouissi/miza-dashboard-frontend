Architecture proposal — no code written

A dashboard shaped like the business, not like the old screens

The legacy pages (/admins, /agents, /clients, /transactions) are discarded as an information architecture. In their place: five business domains — Network, Money, Stock, Grattage, and Reference — mirroring the domain boundaries the backend already enforces (architecture-roadmap.md's Stock/Money split, the Grattage seam, the Spatie permission graph). Every module below is sized to a real cluster of backend routes, not a placeholder file.

01

Design principles

Domain-first IA

Navigation groups by what the business does (move stock, move money, sell grattage), not by what CRUD screen exists. A module exists because a backend domain exists — never the reverse.

One shape per repeating problem

The backend already repeats two shapes constantly: "paginated list with filters" and "draft → lines → irreversible validate." The frontend should build each shape exactly once and configure it per domain, not rebuild it per page.

Permissions as the single source of truth for access

Every nav item, route, and action button reads from the same permission registry the backend already returns at login — closing the sidebar/route drift flagged in the discovery report.

Why Grattage isn't filed under Stock or Money

It's tempting to nest Grattage under either Stock (it consumes commercial stock) or Money (it creates a settlement obligation). The backend explicitly keeps it as neither — architecture-roadmap.md calls the two domains "strictly separated... share no tables and write through no common service," with Grattage/reconciliation as the one coupling point. Giving it its own top-level module — rather than burying it as a tab inside Stock or Money — keeps that boundary visible to whoever builds the UI next, and gives the restock-gate rule (§6) one obvious home instead of two half-homes.

Sidebar rendering rule

A domain group renders only if the user holds at least one permission belonging to it; a sub-item renders only if its own specific permission is present. Same registry as route guards (§10) — one list, two consumers, can't drift.

Breadcrumb

Domain › Sub-module › Entity — e.g. Stock › Allocations › ALLOC-0142. Doubles as the back-navigation target from any detail or workspace page.

05

Top-level modules

Purpose, backend surface, and forward-looking notes for each domain, plus the Overview module that sits above all of them.

Overview home

The only module with no CRUD of its own — a composed read surface over every other domain.

Purpose

Answer, at a glance: is the network healthy, is cash moving, is stock exposure normal, what needs a decision right now (pending cheques, pending deposits, overdue grattage).

Backend APIs
  • GET /admin/dashboard/statistics
  • chartData / recentActivities / agentsOverview — need routing (discovery §6)
  • Composed: /admin/cheques/pending, /admin/depos?status=pending, /admin/grattage-invoices?status=overdue
Future scalability

Built as a widget grid, not a fixed page — each stat tile, chart, or queue is one widget reading one endpoint. A sixth domain later adds a widget, never a redesign. Widget visibility follows the same permission registry as the sidebar (§10).

Network foundation

Admins, Managers, Commercials, Clients — the org graph every other domain references by agent_id / client_id.

Purpose

Manage who can act in the system (internal admins, field managers and commercials) and who they serve (clients). Owns onboarding, status (active/blocked), assignment, and the manager↔commercial hierarchy.

Backend APIs
  • /admin/admins — full CRUD + block
  • /admin/agents — store, managers, commercials, show, update, block, activate, toggle-status, sub-data, grattage-outstanding
  • /admin/clients — index, store, unassigned, statistics, show, update, destroy, status, assign, assign-bulk, unassign, reset-password
Future scalability

The manager→commercial hierarchy is one level deep today. Model the org chart as a generic tree component (parent/children, not hardcoded "manager" and "commercial" columns) so a future third tier (e.g. regional director) is a data change, not a UI rewrite. Client self-registration (OTP) already exists server-side with no admin surface yet — a "pending registrations" queue is a natural first extension.

Money ledger

Cheques, Deposits, Debt Payments — everything that moves cash or grants stock capacity.

Purpose

Turn a cheque into rapped (cash-like) and grattage (stock-capacity) advances; validate cash deposits, including the ones that reconcile a grattage obligation; track each admin's own debt to the company.

Backend APIs
  • /admin/cheques — store, index, pending, show, approve, reject, annuler, agent history
  • /admin/depos — index, store, show, validate, reject
  • /admin/debt-payments — index, store
Future scalability

Deposits already carry a type/status distinction (ordinary vs. Phase 5.10 grattage-reconciliation deposits) — the list view should filter/tab by type from day one, so a third deposit type later is a filter option, not a new page. Pending Cheques and Pending Deposits are both natural instances of the shared ApprovalQueuePage pattern (§8).

Stock ledger

Bons → Allocations → Agent Transfers → Agent Stock Returns — the physical chain from supplier to commercial, plus the parties and products it moves.

Purpose

Track inventory as it descends Supplier → Company (Bon) → Manager (Allocation) → Commercial (Agent Transfer), and back up on return (Agent Stock Return). Every screen here is a variation of "draft, add lines, validate" — validation is irreversible and stock-checked.

Backend APIs
  • /admin/bons — CRUD + validate/cancel + nested lines
  • /admin/allocations — CRUD + validate + nested lines
  • /admin/agent-transfers — CRUD + validate + nested lines
  • /admin/agent-stock-returns — CRUD + validate + nested lines
  • /admin/products — apiResource
  • Companies / Suppliers — no controller yet
Future scalability

All four movement types share one shape and one error-code family (insufficient stock, binding mismatch, role/status drift). Generalize into a single configurable "stock movement" primitive (§9) — a hypothetical fifth movement type becomes a config entry, not a new page class. A cross-cutting "stock ledger" view (filterable by agent, company, or movement type) is a natural addition once all four exist. Companies & Suppliers directory screens are blocked until the backend ships controllers for them — flagged in the discovery report, unchanged here.

Grattage the seam

Retail sales layer — commercials sell stock to clients for cash and incur a same/next-day settlement obligation. Read + reconciliation surface only; creation and settlement are commercial-only / deposit-driven by backend design.

Purpose

Give admins visibility into sales the field made (invoices), which are still owed (outstanding obligation per agent), and which commercials are currently blocked from receiving new stock (restock gate) — without duplicating the commercial's own sale-entry flow.

Backend APIs
  • /admin/grattage-invoices — index, show, cancel (store & settle hard-403 by design)
  • /admin/agents/{id}/grattage-outstanding — obligation + restock-gate state
Future scalability

The restock gate is a rule that other domains must respect (Stock's Allocation and Agent Transfer forms), not a Grattage-only concern — it should ship as a shared hook/component consumed cross-domain (§6), not buried in this module alone. Do not build UI for admin-create or admin-settle: both routes are intentionally disabled server-side (Phase 5.8b / 5.10) and are candidates for deletion on the backend, not for a frontend workaround.

Reference & Settings low churn

Villes, Secteurs, Products — lookup data other domains depend on.

Purpose

Maintain the geography and product catalog that Agents, Clients, Bons, and Allocations all select from. No workflow, no lifecycle — pure reference tables.

Backend APIs
  • apiResource villes / secteurs / products
Future scalability

The proving ground for the generic CRUD-config generator (§9) — lowest risk, fastest to build, and validates the pattern before it's relied on for higher-stakes domains.

07

Reusable component library

Built once, configured per domain. Grouped by the three things the task called out — tables, filters, forms — plus the small set of domain-agnostic primitives everything else leans on.

DataTable

Headless table + sort + row actions + empty/loading/error states. One instance styled once against shadcn, reused by every list in the app.

FilterBar

Search input + typed filter chips (select, date-range, status) declared per resource, synced to the URL query string so filtered views are linkable and survive a refresh.

LineItemsEditor

Product + quantity + unit price row editor with running total. Powers Bon, Allocation, Agent Transfer, and Agent Stock Return line entry — one component, four domains.

ConfirmActionDialog

Typed confirmation for irreversible actions (validate, approve, annuler, cancel) — states the consequence in the resource's own terms, not a generic "are you sure?".

StatusBadge

One semantic-color chip vocabulary (pending / active / validated / settled / overdue / blocked / cancelled) reused across every domain's status column instead of each module inventing its own palette.

MoneyAmount

Tabular-nums currency formatting with a consistent DH/decimal convention — used anywhere an amount renders, from stat tiles to line-item totals.

EntityLink

Mini agent/client/company chip (avatar + name, click-through to detail) — replaces raw IDs anywhere a foreign key would otherwise render as a number.

FileUploadField

Preview + validation wrapper around the image/PDF uploads Agent onboarding and Cheque submission both require.

Timeline

Created → validated/settled → cancelled audit trail, reused on every DetailPage and DraftLifecyclePage that has a status history.

StatCard / ChartCard

Overview widget shells (§5) — a labelled number or a labelled chart in a consistent frame, so new dashboard widgets drop in without new layout code.

PermissionGate

<Can permission="…"> — wraps any button, nav item, or route; renders nothing (or a disabled state with reason) when the permission is absent. See §10.

FormDrawer

Slide-over create/edit form shell (RHF + zod), used for every "quick create" action off a ListPage without leaving the list context.

09

Reusable CRUD structure

Most of §5's modules are the same CRUD shape wearing different field names. Rather than hand-building ~15 near-identical list/detail/form trios, define each resource once as data and let the ListPage/DetailPage/FormDrawer patterns read that definition.

Resource definition (concept, not code)

Per domain resource: its endpoints (list/create/show/update/delete), its table columns, its filter fields, its create/edit form schema, and the permission string gating each action. Villes, Secteurs, Products, and Admins are pure instances of this — zero bespoke page code, only a definition file. Clients and Agents extend the same base definition with the extra pieces they genuinely need (bulk-assign, multi-file onboarding, status-specific actions) rather than being built from scratch.

Where the generic structure stops

Bons, Allocations, Agent Transfers, and Agent Stock Returns are not forced into the plain CRUD generator — they get their own "stock movement" definition (line items, validate/cancel transitions, domain error codes) feeding the DraftLifecyclePage pattern instead. Two generators, not one overloaded one: plain-resource CRUD, and stock-movement lifecycle.

10

Permission architecture

One registry, three consumers — the fix for the sidebar/route drift the discovery report flagged.

1. Registry

A single map of domain → action → backend permission string, mirroring the backend's own App\Authorization\*Permissions classes (BonPermissions, AllocationPermissions, etc.) so the two stay named the same thing on both sides.

2. Route guards

Every route in the router is generated from the registry, not hand-wrapped per route — closing the gap where five legacy routes had no guard at all.

3. UI gating

Sidebar items, action buttons, and form fields all read the same registry through the PermissionGate component (§7) — a permission added to the registry shows up consistently everywhere in one change.

Role model note

The backend has four roles (super-admin, admin, manager, commercial); this architecture is scoped to the two admin-facing roles per the discovery report's open question. If Managers or Commercials are later given dashboard access, the registry and route-guard generator need no change — only new entries — because nothing here hardcodes the two-role assumption structurally.

Domain-driven roadmap

Sequenced by which domains other domains depend on — not by which single page looks best first. Each phase ships a complete domain (list + detail + lifecycle where relevant), not a scattering of half-built pages across several domains.
